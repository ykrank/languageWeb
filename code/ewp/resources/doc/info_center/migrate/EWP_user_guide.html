1 概述

本文旨在指导使用低版本的用户对EWP进行升级，将从系统环境、编译、启动和运行四个方面介绍升级方法。本文不会列举所有需要改动的地方，如想知道所有新旧版本不兼容的修改，请查看各版本升级的参考手册。

2 系统环境

由于EWP5.0之后对依赖的Erlang和Yaws进行了升级，因此在开始修改App之前，我们需要更新系统中的Erlang和Yaws版本，具体步骤可查看参考手册的第二章“基础环境”。

3 编译

在编译过程中，最可能遇到的问题是原来的一些hrl文件或宏找不到了，这是因为EWP5.x版本重新整理了hrl文件，删除一些无用的宏或hrl。具体变动可查看参考手册的第八章“宏”

4 启动

在启动服务之前，我们需要对配置和回调函数做一些修改。具体如下：

4.1 ewp.conf

对于大多数项目来说，需要对ewp.conf做如下修改：

  1. EMP信道证书配置。EWP5.x信道相关配置的格式做了修改，如果仍使用旧版本的配置文件，将无法读取证书。具体修改可查看参考手册的第三章的“安全信道配置”小节。
  2. server配置。EWP5.x整理了提供的服务，因此需要修改server配置。具体修改可查看参考手册的第三章的“server配置”小节。
  3. conf配置项。原conf配置包含的所有配置子项在EWP5.x中均作为单独的配置项，因此需要将conf配置的内容抽取出来。具体修改可查看参考手册的第三章的“conf配置项”小节。
  4. 数据库配置。5.x版本中，数据库连接配置发生了变动，因此使用数据库的一些项目需要修改数据库连接参数。具体修改可查看参考手册的第三章的“数据库配置”小节。
  5. 接口映射配置。在旧版本中，EWP默认的接口映射规则是编写在代码中，为了方便定制，EWP5.x版本将这部分规则改为配置文件控制。具体修改可查看参考手册的第三章的“EWP 接口映射配置”小节。

新版本的ewp.conf配置示例可参考：ewp.conf和security.conf

4.2 回调

开发人员在使用旧版本的EWP时存在对Arg重复解析的问题，造成不必要的性能损失。为了避免这种现象，规范开发，EWP5.x重新整理了大多数回调的输入输出格式，具体修改可查看参考手册的第五章“Callbacks 改动”。 为了让App先启动，我们可以将这些回调按输入参数个数进行修改并返回ok，在运行调试时再修改回调逻辑。例如channel回调list_collection_items(OutputFormat, P, User) 变更为 list_collection_items(OutputFormat, User)。我们可以增加list_collection_items(OutputFormat, User)并返回ok，使其在启动时通过callbacks检查。

4.3 App.app配置

在.app配置文件中，我们只需要添加menu配置来告诉EWP菜单数据的存储方式。修改方法查看参考手册的第三章的“菜单存储类型配置”小节。

5 运行

5.1 加载菜单

EWP5.x版本修改了菜单加载的逻辑，所有菜单数据不在根据配置文件获取，而是在mnesia或数据库中读取，因此在首次启动服务后需要调用菜单加载函数来读取配置文件并加载。具体方法查看参考手册的第九章的“菜单加载”。

此外，菜单配置的位置也从.app文件改为单独的配置文件，具体改动查看参考手册的第三章的“菜单配置”小节。

5.2 配置读取

在5.x版本中，App配置定制配置统一存放在App.conf文件中，读取方法为ewp_conf_util:get_app_conf_value(App, Key)或get_app_conf_value(App, Key, Default)。具体改动查看参考手册的第三章的“app.conf”小节。

为了兼容一些旧版本使用ewp_customize:param(App, Key)获取配置，我们可以按以下方式修改。

假如在ebank.app中配置了以下内容：

{development, [{ebankhost,  "127.0.0.1:4003"},
               {psbc_host, "localhost"},
               {psbc_port, 8081},
               {monitor_ip, "20.13.1.48"},
               {monitor_port, 8273},
               {client_port, 8274},
               {client_flag, all},
               {persistent, false},
               {conn_size, 100},
               {conn_timeout, 5000},
               {send_timeout, 5000},
               {recv_timeout, 50000}]}.

在ebank启动时，可以增加以下代码：

    ConfList = ewp_conf_util:get_app_conf_value(ebank, development),
    ets:insert(ebank, ConfList),

这样便可以使用ewp_customize:param(App, Key)方法获取配置，而不用修改代码。注意的是，示例中的代码只能获取到development的配置。因此即使切换到production环境获取的配置依然为development配置。

5.3 参数获取

在5.x版本中，大部分回调函数的参数中不再包含Arg和P参数。可以通过Arg = ewp_params:get_args()和P = ewp_params:get_params()来获取这两个参数。需要注意的是，通过ewp_params:get_args()获取到的Arg参数后，再调用P = ewp_params:from_yaws_arg(Arg)是无法获取所有请求参数的。

另外在新版本中，除了app，o，"id"和"tranCode"外，其他请求参数都可以通过ewp_params:param/1方法获取。这可能造成一些不兼容的情况，如在邮储后台的代码中存在以下逻辑

Root = ebank_utils:handle_internal_form(
    [{"tranCode", ewp_params:param("tranCode")} |
     dict:to_list(P)], undefined)
在旧版本中，由于请求的url中不含有tranCode而是在请求参数中包含，所以{"tranCode", ewp\_params:param("tranCode")}最后将变成{"tranCode", undefined}，经过判断可将其过滤。而在新代码中，由于请求参数包含tranCode，将导致[{"tranCode", ewp\_params:param("tranCode")} | dict:to\_list(P)]的结果包含两个相同值的tranCode。如果将其作为cs模板的输入，那么将无法正常取值。 更多关于取值方式的改动，可查看参考手册的第四章“取值接口改动”。 ### cs模板 #### 过滤换行 cs模板在3.x版本中的换行符不会被过滤，在4.x版本中的换行符会被删除。但在5.x版本中，换行符会被替换成空格，这可能导致一些页面无法正常显示。此时可以修改ewp.conf的cs\_enter\_filter配置，为0时不过滤换行，为1时删除换行。 #### channel/run模板输入 EWP5.x中删除了channel/run流程中cs模板输入的ewp_host和time值。如需要获取两个值可以在transform返回的模板输入中添加{ewp\_host, ?arg("host")}和{time, ewp\_time:to\_iso8601(erlang:localtime())}。 ### regexp API兼容 由于Erlang R15B版本中移除了regexp模块，并推荐使用re模块函数来实现相同功能。在一些App代码中使用了regexp模块的API，需要替换使用re模块实现兼容。例如： 
    case regexp:match(ReceiverContactPhone, "^[0-9]+$") of
        nomatch -> throw("您输入的收货人联系电话格式不正确，请重新输入。");
        _ -> go_on
    end,
    可以替换为
    case re:run(ReceiverContactPhone, "^[0-9]+$") of
        nomatch -> throw("您输入的收货人联系电话格式不正确，请重新输入。");
        _ -> go_on
    end,



    {ok, Var, _} =regexp:gsub(Str, Replace, NewStr),

    可替换为

    Var = re:replace(Str, Replace, NewStr, [global, {return, list}]).



    {match, List} = regexp:matches(Pin, L),

    可替换为

    {match, List0} = re:run(Pin, L, [global]),
    List = [{Start + 1, Len} || [{Start, Len}|_] <- List0],

5.4 异常处理

EWP5.x对App抛出的异常处理逻辑进行了优化，使用error_code_service服务对App定义的异常进行统一管理。为了兼容旧版本的处理方式，我们定义error callback来处理抛出的异常。以邮储银行后台为例，在邮储旧版本后台的交易代码中，throw一个字符串Msg后，EWP会调用ewp_render_util:render_error_xml(Msg)将错误信息组装成XML。升级后，为了兼容该逻辑，我们可以按一下步骤修改：

首先修改.app文件，添加callbacks配置如下。添加后代码执行出现异常时将由ebank_error:handle/1处理

...

{callbacks, [{error, {ebank_error, handle}}]},

...

然后添加ebank_error.erl模块，编写如下代码：

-module(ebank_error).

-export([handle/1]).

handle({throw, [H|_] = Msg}) when is_list(Msg) andalso is_integer(H) ->
    ewp_render_util:render_error_xml(Msg);
handle({throw, Error}) when is_binary(Error) orelse is_atom(Error) ->
    ewp_render_util:render_error_xml(Error);
handle({throw, Error}) ->
    error_code_service:dispatch_error(Error);
handle(_) ->
    error_code_service:dispatch_error("unknown error").

编译运行后即可完成兼容。

5.5 回调方法兼容

为了减少回调方法兼容改动的工作量，可以保留原回调函数，定义新回调函数并获取原回调函数需要的参数后调用原回调函数。

以邮储ebank_channel回调修改为例，具体修改为：

-export([role_auth/2,
         list_collection_items/3,
         get_cache_mapping/0,
         transform/4,
         gen_query_request/5,
         request/6]).
    替换为
-export([role_auth/2,
         list_collection_items/2,
         get_cache_mapping/0,
         transform/3,
         gen_query_request/2,
         request/4]).

list_collection_items(Format, P, User) ->
    CsKey = ewp_channel_util:get_cskey(list, Format),
    替换为：
list_collection_items(Format, User) ->
    CsKey = ewp_channel_util:get_cskey(list, Format),
    P = ewp_params:get_params(),

gen_query_request(Channel, Url, AppendedParams, _P, _User) ->
    替换为：
gen_query_request(Channel, Url) ->
    AppendedParams = ewp_params:filter_params_to_url(),

request(Channel, _FormUrl, P, _User, _A, TranCode0) ->
    替换为：
request(Channel, _FormUrl, _User, TranCode0) ->
    P = ewp_params:get_params(),

追加回调方法：
transform(Format, Res, Channel) ->
    Arg = ewp_params:get_args(),
    P = ewp_params:get_params(),
    Params0 = dict:fold(fun(Key, Val, Acc) -> 
                            lists:concat([Acc, "&", Key, "=", yaws_api:url_encode(ewp_str_util:to_list(Val))]) 
                        end, "", P),
    NewParams = 
    case Params0 of
      [] ->
        [];
      _ ->
        tl(Params0)
    end,
    transform(Format, Res, Channel, Arg#arg{clidata=list_to_binary(NewParams)}).
