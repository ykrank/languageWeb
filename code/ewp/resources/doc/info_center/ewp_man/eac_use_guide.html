1 概述

在EWP 5.3.7的版本中，优化了EWP管理平台，将管理平台从EWP中分离出来，并将其分成两部分：EAC（页面控制台）和ADMIN（服务插件）。如图1-1所示：

          图1-1 管理平台优化后的变化

2 EAC（EWP Admin Console）

2.1 EAC介绍

AC是从EWP中分离出的页面控制台，他负责的功能：

  1. 页面展现和管理。EAC为管理后台提供前端的页面展现，而且能够整合所有的APP的管理后台，只需通过配置文件，就能够访问不同APP的管理后台。
  2. 用户管理功能。支持非管理员的注册以及权限管理。
  3. 扩展功能。EAC支持内部自定义功能扩展。

2.2 EAC部署

EAC的部署方法与其他APP（ebank、emas）的部署方法相似，操作步骤为：

  1. 获取代码
  2. 编译
  3. 配置
  4. 启动

2.2.1 获取代码

将代码获取到本地： EAC的代码路径：https://192.168.64.239/hg/proj/emp/ewp_admin 提示：获取代码前，请先申请代码权限

2.2.2 编译

从服务器上获取EAC的代码后，放到本地

  1. 在ewp_admin路径下，执行./configure如需要调试log，则可加上debug指令./configure --with-debug
  2. make

2.2.3 配置

EAC有两个配置文件：eac.conf 和 admin_menu.conf。

2.2.3.1 eac.conf

文件路径： ewp_admin/config/eac.conf 参考样例：

{app_urls, [{"emas", "192.168.64.127:4443"},
            {"ebank", "192.168.64.127:4000"},
            {"unit-test", "ebank", "192.168.64.127:4009"}]
}.

配置方式： EAC访问APP的地址配置，只有正确的配置访问APP

{app_urls, [{AppName, AppUrl},
         {AppAlias, AppName, AppUrl}]
}.

AppName：应用名，参数类型：string() AppAlias：应用别名，如"unit-test"，使用别名"unit-test"登录，但是实际使用的应用名为"ebank"，参数类型：string() AppUrl：访问应用的URL，参数类型：string() 特别解释：AppUrl -> "IP:Port" 　　IP：访问应用的服务器IP地址 　　Port：参考访问应用yaws.conf中的port配置

2.2.3.2 admin_menu.conf

文件路径：ewp_admin/config/admin_menu.conf 参考样例（图2-1）：
          图2-1 admin_menu.conf示例

配置方式如下： 一级菜单 

{pre_menu,[{id,"pre_menu_id"},
            {name,"pre_menu_name"},
            {seq,1}]}.

id：一级菜单的唯一ID(全局唯一) name : 菜单名 seq :菜单排列序号 二级菜单 

{sec_menu,[{id,"sec_menu_id"},
            {name,"sec_menu_name"},
　　         {pre_menu,menu_id},
            {index,"html_page"},
              {param_types, [{func1, eac_bag}, {func2, eac_file}]}
            {seq,1}]}.

id：secmenu的ID，此id也对应着请求的Mod名（全局唯一） name：sec_menu的名称 pre_menu：指定关联的pre_menu 的id index：使用的html页面 param_types：上送参数种类的集合 seq：菜单排列序号 param_types描述： param_types是管理平台上送参数种类的集合。也是功能接口的映射规则。 配置方式：{param_types, [{func, type}]} func：调用方法名 type : 调用方法参数类型（eac_set | eac_bag | eac_file | ewp_normal | ewp_file） EAC功能扩展标识"eac"：

  • eac_set：将yaws arg记录存储的请求参数转换成请求参数字典，作为接口参数请调用 Mod:Func(dict())。
  • eac_bag：对yaws arg不做处理，直接请求接口模块 Mod:Func(arg())，如果请求参数会有一对多的情况，可以使用该配置，在回调模块中使用ewp_param:bag_from_yaws_arg(Arg)进行请求参数转换。
  • eac_file：对 Content-Type为"multipart/form-data"(一般是上传文件操作)的请求做处理，将文件写到ewp目录下的public/data/upload/FileName，并且将yaws arg记录存储的请求参数转换成请求参数字典，作为接口参数请调用 Mod:Func(dcit())。

ADMIN功能插件扩展标识"ewp_"：

  • ewp_file：与eac_file类似，将上传文件数据整合后，通过HTTP请求到ADMIN插件侧，由ADMIN插件对文件数据进一步逻辑处理。
  • ewp_normal（默认值）：除文件类型以外的类型，将数据通过HTTP请求到ADMIN插件侧，由ADMIN插件进行分类、逻辑处理。

paramtypes的默认值为“ewp_normal”。如果该Mod没有配置param_types，那么该Mod所有的Func的参数类型都为“ewp_normal”。 ※带有"eac"标识的扩展功能是EAC内部功能，不与ADMIN插件交互。

2.2.4 启动

在ewp_admin路径下，执行脚本./iewp

2.3 EAC功能扩展

EAC支持内部功能扩展，方法为：在完成页面和功能模块后，再通过一定的配置就可以做到EAC的功能扩展。

2.3.1 EAC的映射逻辑

想要通过页面的请求，完成后台功能的调用，需要通过eac_controller模块进行解析。而eac_controller解析的规则，是通过配置文件:admin_menu.conf来完成的。 admin_menu.conf配置示例：

{pre_menu,[{id,      "eac_demo"},
           {name,    "DEMO"},
           {seq,     10}
           ]}.
{sec_menu,[{id,          "demo_eac"},
           {name,        "EAC_DEMO"},
           {pre_menu_id, "eac_demo"},
           {index,       "eac_demo.html"},
           {param_types, [{"eac_set_test", eac_set},
                          {"eac_bag_test", eac_bag},
                          {"eac_file_test", eac_file}]},
           {seq,         1}
           ]}.

下图表示EAC的逻辑映射关系：

                图2-2 EAC逻辑映射关系
　　　① http://host:port/**demo_eac/eac_set_test**
　　　② http://host:port/**demo_eac/eac_bag_test**
　　　③ http://host:port/**demo_eac/eac_file_test**
EAC的实际扩展Mod名，会在Mod前追加前缀"eac"，即调用EAC扩展功能，实际Mod为"eac_Mod"。根据admin_menu.conf配置内容，映射结果：
　　　① -> eacdemo_eac:eac_set_test
　　　② -> eacdemo_eac:eac_bag_test
　　　③ -> eacdemo_eac:eac_file_test

2.3.2 EAC扩展实例

  1. 编写好EAC功能模块：eac_demo_eac.erl，内部实现方法：eac_set_test/1、eac_bag_test/1、eac_file_test/1。
  2. 编写好html页面eac_demo.html。页面上调用接口的Url为：demo_eac/eac_set_test、demo_eac/eac_bag_test、demo_eac/eac_file_test。
  3. 将写好的功能模块放到EAC的代码中：ewp_admin/src，页面放到EAC页面资源路径下：ewp_admin/public/admin。

  4. 配置admin_menu.conf

    {pre_menu,[{id,      "eac_demo"},
        {name,    "DEMO"},
        {seq,     10}
        ]}.
{sec_menu,[{id,          "demo_eac"},
        {name,        "EAC_DEMO"},
        {pre_menu_id, "eac_demo"},
        {index,       "eac_demo.html"},
        {param_types, [{"eac_set_test", eac_set},
                       {"eac_bag_test", eac_bag},
                       {"eac_file_test", eac_file}]},
        {seq,         1}
        ]}.

  5. 编译EAC，重启服务，登录管理平台，点击“EAC_DEMO”进入扩展实例页面。

3 ADMIN插件

3.1 ADMIN插件介绍

ADMIN插件提供的是EWP管理后台逻辑处理功能。它依赖于EWP启动，处理EAC通过HTTP传输的请求，再通过HTTP将处理结果返回给EAC。可支持功能扩展。

3.2 ADMIN插件部署

3.2.1 安装

ADMIN插件已预装在EWP上，安装EWP 5.3.7版本后的安装包即可。

3.2.2 配置

3.2.2.1 ewp.conf

ADMIN插件是在EWP启动中进行加载的，所以，需要在相应的APP的ewp.conf配置文件中追加配置选项。
文件路径：APP/config/ewp.conf （APP：ebank、emas、unit-test……） 参考样例：

{ewp_addon, [
　　{admin,[
        %default is ewp_install_dir/addon/admin/config/admin_url_map.conf
        %{map_conf, "/var/www/apps/APP/config/admin_url_map.conf"}
    ]}
]}.

配置方式： {ewp_addon,[{AddonName, Options}]}.

AddonName：插件的名称，参数类型：atom()
Options：插件属性。
Options = [Option]
Option = {map_conf, FilePath :: string()}
map_conf : ADMIN映射规则配置文件路径，可指定文件路径，如不配置此属性，
则使用默认路径：ADMIN/config/admin_url_map.conf
3.2.2.2 yaws.conf

每一个APP的ADMIN插件的service是不一样的，所以在yaws.conf中需要添加插件的配置。 文件路径：APP/config/yaws.conf 参考样例：

<server admin>
    port = 4000
    listen = 0.0.0.0
    docroot = public/admin
    appmods = </admin, admin_controller></, error_url>
    auth_skip_docroot = true
    <opaque>
        admin = true
    </opaque>
</server>

server的名字为 admin port :监听的端口（这里的端口同时也用在eac.conf的URL配置中）。 docroot :server使用的根目录，一般用来指定页面所存放的位置。 appmods :yaws回调模块。 auth_skip_docroot :是否跳过寻找docroot文件夹的.yaws_auth文件 opaque/admin = true : 管理平台使用的标识。

3.2.2.3 admin_url_map.conf

ADMIN插件有一个重要的配置文件：admin_url_map.conf。它是ADMIN插件功能的接口映射规则。此配置文件路径是在APP的ewp.conf中配置的，在上面(配置文件:ewp.conf)已经说明。 文件路径： APP/config/admin_url_map.conf
默认配置路径： Ewp_install_dir/addon/admin/config/admin_url_map.conf
参考样例：

{url_map,[
    {
        {"log"},log,[{param_type, bag}]
    },
    {
        {"demo_admin"},demo_admin,[]
    },
    {
        {"demo_admin", "admin_bag_test"},demo_admin,[{param_type, bag}]
    },
    {
        {"demo_admin", "admin_file_test"},demo_admin,[{param_type, file}]
    }
    ]
}.

配置方式： {url_map, [{UrlPrefix, Mapping, Options}]}. UrlPrefix：URL的相对路径，例如："/demo_admin/admin_bag_test" 结构：{MOD}、{MOD, FUNC} 类型：MOD :: string() FUNC :: string() Mapping：映射模块，参数类型：atom() Options: 接口配置选项，参数类型：[option] = [{param_type, Type()}] Type() = set | bag | file param_type描述： 与EAC的"param_types"基本相同。

  • set：将yaws arg记录存储的请求参数转换成请求参数字典，作为接口参数请调用 Mod:Func(dict())。
  • bag: 对yaws arg不做处理，直接请求接口模块 Mod:Func(arg())，如果请求参数会有一对多的情况，可以使用该配置，在回调模块中使用ewp_param:bag_from_yaws_arg(Arg)进行请求参数转换。
  • file: 对 Content-Type为"multipart/form-data"(一般是上传文件操作)的请求做处理，将文件写到ewp目录下的public/data/upload/FileName，并且将yaws arg记录存储的请求参数转换成请求参数字典，作为接口参数请调用 Mod:Func(dcit())。

param_type的默认值为“set”。如果该{MOD}没有配置param_type，那么该Mod除了{MOD, FUNC}以外所有的Func的参数类型都为“set”。

3.3 ADMIN插件功能扩展

ADMIN插件也功能扩展，与EAC的思想相同，方法为：完成页面和功能模块后，再通过配置一些配置，完成功能扩展。

3.3.1 ADMIN插件映射规则

从页面请求，到调用ADMIN内部接口。需要经过eac_controller和admin_controller两个模块的解析。eac_controller根据“admin_menu.conf”解析后，通过HTTP，继续请求到“admin_controller”，由admin_controller模块解析“admin_url_map.conf”完成最终的映射逻辑。 admin_menu.conf配置示例：

{pre_menu,[{id,      "eac_demo"},
           {name,    "DEMO"},
           {seq,     10}
           ]}.
{sec_menu,[{id,          "demo_admin"},
           {name,        "ADMIN_DEMO"},
           {pre_menu_id, "eac_demo"},
           {index,       "admin_demo.html"},
           {param_types, [{"admin_set_test", ewp_normal},
                          {"admin_bag_test", ewp_normal},
                          {"admin_file_test", ewp_file}]},
           {seq,         2}
           ]}.

admin_url_map.conf配置示例：

{url_map,[
    {
        {"demo_admin"},demo_admin,[]},
    {
        {"demo_admin", "admin_bag_test"},demo_admin,[{param_type, bag}]},
    {
        {"demo_admin", "admin_file_test"},demo_admin,[{param_type, file}]}
    ]}.

下图表示ADMIN的逻辑映射关系：

                图3-1 ADMIN的逻辑映射关系
　　　① http://host:port/**demo_admin/admin_set_test**
　　　② http://host:port/**demo_admin/admin_bag_test**
　　　③ http://host:port/**demo_adminadmin_file_test**
　　　ADMIN实际使用的模块，是在secmenu的id追加前缀"admin"，即调用ADMIN扩展功能，实际使用的Mod为"adminMod"。根据admin_menu.conf、admin_url_map.conf配置内容，映射结果：
　　　① -> admindemo_admin:admin_set_test
　　　② -> admindemo_admin:admin_bag_test
　　　③ -> admindemo_admin:admin_file_test

3.3.2 ADMIN插件扩展实例

  1. 编写好ADMIN功能模块：admin_demo_admin.erl，内部实现方法：admin_set_test/1、admin_bag_test/1、admin_file_test/1。
  2. 编写好html页面admin_demo.html。页面上调用接口的Url为：demo_admin/admin_set_test、demo_admin/admin_bag_test、demo_admin/admin_file_test。
  3. 将编写好功能模块放到ADMIN的代码中：APP/src，页面放到EAC页面资源路径：ewp_admin/public/admin。

  4. 配置admin_menu.conf (接上面EAC扩展)

    {pre_menu,[{id,      "eac_demo"},
        {name,    "DEMO"},
        {seq,     10}
        ]}.
{sec_menu,[{id,          "demo_admin"},
        {name,        "ADMIN_DEMO"},
        {pre_menu_id, "eac_demo"},
        {index,       "admin_demo.html"},

-module(eac_demo_eac).

-include("eac.hrl").

-export([eac_set_test/1,
         eac_bag_test/1,
         eac_file_test/1]).

-define(UPLOAD_DIR, filename:join([?ewp_conf(ewp_dir, ""),"public/data/upload/"])).

%% ====================================================================
%% API functions
%% ====================================================================
eac_set_test(_P) ->
    Param = ?param("param_set", []),
    Str = {obj, [{"message",
                 list_to_binary("ok")},
                 {"param", list_to_binary(Param)}]},
    JsonDec = ewp_json:encode(Str),
    ewp_render_util:render_json(JsonDec).

eac_bag_test(Arg) ->
    P = ewp_params:bag_from_yaws_arg(Arg),
    Name = ewp_params:get("name", P, "undefined"),
    Lists = [List ++ " " || List <- ewp_params:get("select_id", P, [])],
    Str = {obj, [{"message",
                 list_to_binary("ok")},
                 {"name", list_to_binary(Name)},
                 {"param", list_to_binary(Lists)}]},
    JsonDec = ewp_json:encode(Str),
    ewp_render_util:render_json(JsonDec).

eac_file_test(P) ->
    ewp_params:merge_params(P),
    UploadFilePath = filename:join([?UPLOAD_DIR, ?param("file")]),
    ?ewp_msg("~p~n", [UploadFilePath]),
    Str = {obj, [{"message",
                 list_to_binary("ok")},
                 {"path", list_to_binary(UploadFilePath)}]},
    JsonDec = ewp_json:encode(Str),
    ewp_render_util:render_json(JsonDec).
