Session操作

1 概述

EWP Session是基于Erlang ETS（一种内存数据库）实现的会话管理工具，主要用来临时保存和快速存取交易数据。由于使用内存作为存储，Session具有以下特点：

  1. 数据存取快速。由于对内存读写，Session数据的存取速度要快于通过磁盘IO和网络IO实现存取功能，如数据库和Web服务。
  2. 数据量小。受内存空间限制，Session存储的数据量远不如持久化存储工具，如数据库。
  3. 临时存储。为限制过期的Session对内存的占用，EWP会定时清理过期Session，此时该Session中的数据将被全部清除。同时，Session使用的内存空间会随着EWP进程关闭而释放。基于以上两点，可以知道Session只能作为一种临时数据的存取工具。

基于以上特性，EWP Session多用在以下两种场景中：

  1. 存储临时数据。如用户在线状态、交易验证码等。
  2. 缓存数据。对于一些静态的，通过数据库或Web服务获取到的数据，可以缓存在Session中，从而提高读取效率。如用户ID、姓名、手机号等。

2 使用示例

EWP的session_service模块提供了Session操作的API，API说明可参考EWP API说明

我们以登录为例，在登录逻辑中，我们创建一个Session，并将用户名写入到Session中。

首先创建一个简单的登录页面，代码如下：

<body>
    <form action="http://localhost:4002/demo/login">
        用户：<input type="text" name="user" length="10" /><br/>
        密码：<input type="password" name="passwd" length="10" /><br/>
        <input type="submit" value="登录"/>
    </form>
</body>

点击登录后，浏览器将访问demo/login接口，我们需要实现此接口逻辑并添加URL映射关系。修改app文件的controllers配置，添加以下URL映射关系：

    {"demo", demo, [{decrypt, false}, {verify, false}]},

添加后，demo/login的请求将由demo_controller.erl模块的login函数处理。接下来我们创建demo_controller模块并实现login函数，代码如下：

-module(demo_controller).

-export([login/1,
         lookup/1]).

-include("ewp.hrl").

login(Arg) ->
    ewp_params:save_yaws_arg(Arg),
    SessionId = session_service:new_session([{"user", ?param("user")}]),
    Res = login_res(),
    ewp_render_util:render_html(Res, [session_service:make_cookie_str(SessionId)]).

login_res() ->
    "<body>"
        "登录成功<br/>"
        "<form action='/demo/lookup'>"
            "<input type='submit' value='查询用户名'/>"
        "</form>"
    "</body>".

lookup(Arg)->
    {_, Session} = session_service:read(Arg),
    UserName = o(Session, "user"),
    ewp_render_util:render_text("用户名为" ++ UserName).

完成后编译重启服务。打开写好的登录页面，输入用户名和密码登录后可以看到以下内容：

登录成功

点击“查询用户名”按钮后可以看到以下内容：

用户名为1231

3 示例讲解

在登录流程中，login函数中调用了session_service:new_session/1方法。改方法将创建一个Session并保存指定的数据并返回Session标识ID，数据格式为[{Key, Value}]。示例中保存了登录的用户名并以"user"为key值。在返回登录成功页面时，我们调用了session_service:make_cookie_str/1方法创建了一个response header。这个header将设置浏览器的cookie，使得SessionId保存在cookie中。保存后，EWP就可以在浏览器后续的请求的cookie中获取到SessionId，这样我们在后续的逻辑代码中变可以根据SessionId获取到登录时创建的Session。

点击“查询用户名”按钮后，浏览器请求了demo/lookup接口。根据配置的映射关系，该请求将进入demo_controller:lookup/1函数处理。在lookup函数中，我们首先使用了session_service:read/1方法。该方法将从浏览器请求的cookie中获取SessionId并读取对应的Session。通过o函数查找Session中的"user"值便可以获取到登录时保存的用户名。

4 小结

本文中，我们通过一个简单的登录示例对EWP Session有了初步的认识。在EMP开发中，Session操作会比示例中的内容更加方便。在ERT设备与EWP服务完成握手时就会创建Session，因此我们不用考虑在哪个业务中初始化Session。在channel逻辑中，通过?session("user")宏便可以读取到Session中保存的"user"值，如果需要获取整个Session数据，通过{SessionId, Session} = session_service:read()方法即可，无须获取当前请求的yaws arg记录。
