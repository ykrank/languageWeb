错误服务使用手册

1 概述

EWP从4.3起就引入了error_code_service服务，用以提供错误码统一处理机制．由于缺乏完整的接口文档说明及后续的一些bug fix， 开发者在使用错误码时经常会感到困惑．编写本文档的目的正是为了澄清其内部机制及相关定义，以及描述api说明，使用场景等．

此处假定你也了解本文档前面基础开发篇。

2 约定

为了描述方便，我们约定以下名称：

  • ECS
    error_code_service，提供错误码统一处理机制的整个功能模块．

  • APP
    基于ewp框架的项目应用，作为ECS的使用者．

  • ERROR
    一条错误码信息记录，包含多个描述自身的属性，其中至少包含一个可唯一标识自己的关键属性．

  • ERRKEY
    唯一标识ERROR的关键属性．

  • HANDLER
    当发生某种错误(ERRKEY)时执行的特定动作．

  • ERRSTORE
    存储ERROR集合的地方，被ECS根据ERRKEY从中查找ERROR．

3 内部机制

ECS只提供当发生某种错误(ERRKEY)时执行预先设定的动作(HANDLER)的机制， 至于错误的存储ERRSTORE及基于ERRKEY如何获取HANDLER都可由使用者决定． 因此，在ECS的初始化阶段需要告知ECS如何获取HANDLER．

在ECS中，HANDLER包含在ERROR中，因此，也就是要告知ECS如何通过ERRKEY获取ERROR． ERROR定义为以下 记录(ewp/include/models/error_code.hrl)：

-record(err, {
    code,          %% 错误码，一般用作唯一标识
    type,           %% 错误类型，用于错误分类的描述性信息
    cust_msg,   %% 错误信息，呈现给最终用户的文本信息
    dev_msg,    %% 错误信息，呈现给开发者的文本信息
    handler       %% 发生此错误时被执行的动作
})

例如，APP可以把ERROR存放到ets当中，这个时候ets充当ERRSTORE， 然后通知ECS以ets:lookup/2的方法来获取ERROR．也可以把ERROR存放数据库当中， 这个时候数据库充当ERRSTORE，然后通知ECS以select查询的方式来获取ERROR．

由于ECS支持多app错误码管理，因此，每一个ERROR都对应到一个APP．当发生一个错误时，ECS知晓并获取设定的ERROR，最后执行返回．

总体来说，ECS支持的功能有：

  • 多app支持，即每个app都可以单独定义错误码，它们互不冲突．
  • 自定义错误的存储方式及查询方式．
  • 自定义错误码结构，但最终必须转型到ERROR上．

4 实现及接口说明

4.1 初始化

error_code_service提供了默认的基于ets存储的错误码查找管理方式， 如果APP不需要自定义存储则可以直接使用这种默认方式． 使用这种方式需要用到以下任一接口：

load_from_file(App::atom(), FilePath::string())．%% 从指定文件中加载ERROR．
load_from_records(App::atom(), Records::list())．%% 从一个ERROR的列表中加载．

从文件中加载其实是将文件中的内容读取成ERROR列表，最后再调用load_from_records完成． ERROR结构中只有code和cust_msg是必须的．在文件中的配置方式如下：

%% {app, [
%%   {"code", "cust_msg"},
%%   {"code", "type", "cust_msg", "dev_msg", "handler"}
%% ]}
{ebank, [
    {"ER001", "SYSTEM ERROR", "System Exit", "kernel panic: no init found", undefined},
    {"ER002", "NETWORK ERROR", "Connection Timeout", "network card not detected", undefined},
    {"ER003", "Other Error"}
]}.

或者直接以ERROR列表的方式读取：

load_error_conf() ->
    Error_Rec = [#err{code     = "CM_0000",
                      type     = 'app_error',
                      cust_msg = "ebank_error",
                      handler  = {page, "error"}}],
    error_code_service:load_from_records(ebank, Error_Rec).

在ERROR结构中是可以设置HANDLER的．HANDLER有以下几种：

{callback, {Mod, Fun}}．设置Mod:Fun/2函数为HANDLER．
{page, CSKEY}．设置CS模板进行处理．

其中，callback类型的HANDLER函数接收的两个参数的类型分别为#err_app{}和#err{}． 这种类型的HANDLER将直接执行．

page类型则是一种预定行为的HANDLER，它将把ERROR与名为CsKey的模板直接交给cs engine去处理．下面的代码便是ECS中render过程， 从中可看出在模板中可获取的变量．

%% 这里的Tmpl是将CsKey补app参数前缀和o参数后缀后的名称．
%% 例：若CsKey="error"，o=i，app=fake，
%%    则Tmpl为"fake_error_i"或"error_i"
render_error_page(#err_app{o=O}, Error, Arg, Tmpl) ->
    Input = [{time, ewp_time:to_iso8601(erlang:localtime())},
             {host,ewp_yaws_util:get_host()},
             {cust_msg, Error#err.cust_msg},
             {dev_msg, Error#err.dev_msg},
             {type, Error#err.type},
             {code, Error#err.code}|Arg],
    %?ewp_log("render error with input:~p~n", [Input]),
    Content = cs_api:render(Tmpl, Input)

如果需要自定义错误码存储方式，则需要通知ECS如何基于ERRKEY来查找ERROR，在ECS中实现为一个search函数：

set_search_fun(App:atom(), F)．F = {Mod, Fun} | fun Mod:Fun/1 | fun Fun/1．

这里的F即为设置的基于ERRKEY查找ERROR的回调函数．它接收一个ERRKEY作为参数． 在成功找到ERROR时返回{true, ERROR}，否则返回false．例如：

find_err(Key) ->
    #err{code=Key, cust_msg="cust"}.

init_app_errors() ->
    ok = error_code_service:set_search_fun(ebank, fun find_err/1).

4.2 设置APP默认的HANDLER

我们已经知道，每个ERRKEY发生时将会执行对应ERROR中的handler项设置的HANDLER， 但这个HANDLER不是必须设置的．因为有时候并不需要对每个ERRKEY去单独设置HANDLER， 这个时候就可以设置默认的HANDLER：

set_default_handlers(App::atom(), DH::default_handlers())．

default_handlers()是一个record类型，定义在ewp/include/models/error_code.hrl中，其中可设置：

  • page
    设置page类型的默认HANDLER，其值为字串形式的CS模板名．
  • callback 设置callback类型的默认HANDLER，其值为{Mod, Fun}．接收的两个参数的类型分别为#err_app{}和#err{}．

4.3 错误码的分发

APP需要在它的代码中发生"错误"时分发出ERRKEY，这时可用以下接口：

dispatch_error(Key::term()))．%% ECS将自动分析分发时的APP环境信息．
dispatch_error(ErrApp::err_app(), Key::term())．%% 手动指定APP环境信息．

这里的Key可为以下几种形式之一：

  • ERRKEY
    直接分发出错误，查找HANDLER的判定顺序是： ERROR中有没有设定HANDLER-->APP有没有设定默认的HANDLER-->使用ewp的默认page类型的HANDLER．

  • {page, ERRKEY}
    查找HANDLER的判定顺序是： ERROR中有没有设定page类型的HANDLER-->APP有没有设定默认的page类型的HANDLER-->使用ewp的默认page类型HANDLER.

  • {page, ERRKEY, Arg}
    同上，只是在CS engine处理时模板输入多了Arg参数．

  • {callback, ERRKEY}．
    查找HANDLER的判定顺序是： ERROR中有没有设定callback类型的HANDLER-->APP有没有设定默认的callback类型的HANDLER-->使用ewp的默认callback类型HANDLER.

  • {callback, ERRKEY, Arg}．
    同上，只是在回调函数的输入参数列表中多了Arg参数．

需要注意的是，ewp自身也会作为一个APP来加载ewp的错误码，并交由ECS管理，这个加载过程在ewp的启动阶段．

另外，除了调用dispatch_error接口来分发错误外，当你的流程在ewp:out/1之内，还可以直接使用throw/1的方式． throw的优势是在语言层面将直接返回到ewp:out/1．

4.4 错误码的销毁

在APP结束时可能需要做一些清理工作．当APP使用了ECS提供的默认错误码存储及查找功能时，还需要调用以下接口 来清除之前加载的错误码．

unload_app_errors(App::atom())．

5 使用场景

当APP中需要为一些业务逻辑错误设定对应的展示结果时，可以考虑使用ECS． ECS在于统一及规范化这类场景的流程．

如果错误码的数量非常巨大，这个时候选择ECS默认的基于ets存储机制的方式就可能有问题，应该自定义存储方式了．

ECS中``没有规定ERROR中的code字段必须为key，只是在其提供的默认存储机制里是这样实现的． 所以，当需要其它字段作为key来查询ERROR`时可以通过自定义存储来实现．

6 示例

fake项目的程序猿开始头疼了，因为BT的boss要求从一个文件中读取错误信息并展示，关键是这个文件是另一个系统实时生成的， 我们不能简单的拷贝这份文件的内容再加载到ECS中． 这份文件名为error_codes，其部分内容如下：

ap001  登陆错误
ap002  余额不足

还好这里面的内容是每行一个Key-Value对．由于ECS可以自定义存储方式，然后...

init_fake_errors() ->
    Pid = erlang:spawn(?MODULE, find_service, ["error_codes"]),
    true = erlang:register(find_service, Pid),

    DH = #default_handlers{callback = {?MODULE, show_msg}},
    ok = error_code_service:set_default_handlers(fake, DH),
    ok = error_code_service:set_search_fun(fake, fun find/1).

find(Key) ->
    find_service ! {self(), search, Key},
    receive
        {data, {Code, CustMsg}} ->
            {true, #err{code = Code,
                 cust_msg = CustMsg}};
        _ ->
            false
    end.

find_service(FileName) ->
    receive
        {From, search, Key} when is_list(Key), is_integer(hd(Key)) ->
            {ok, C} = file:read_file(FileName),
            Option = [{capture, all_but_first, list}, multiline],
            case re:run(C, "^"++Key++"\\s+"++"(\\w+)$", Option) of
                {match, [Msg]} ->
                    From ! {data, {Key, Msg}};
                _ ->
                    From ! no_data
            end,
            find_service(FileName);
        stop ->
            stop;
        _ ->
            find_service(FileName)
    end.

show_msg({#err_app{name=fake}, #err{cust_msg = Msg}}) ->
    io:format("  ++show msg: ~p~n", [Msg]),
    Msg.
