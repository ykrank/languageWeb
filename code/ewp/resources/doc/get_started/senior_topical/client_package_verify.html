1 服务器配置文件修改

security.conf添加字段client_verify

    {client_verify,[

    {flag,false}, %防篡改校验开关，true为打开，

    {client_verify_file,"public/security/client_file"}, %校验文件摘要存放路径（要有读写权限）

    {verify_version,[{1, 2},{1, 3}]}, %需要进行校验的信道版本号

    {pass_version,[{1, 0}]}, %直接通过校验的信道版本号（此字段只提供版本升级的过度时期使用，以防篡改者篡改信道版本号来跳过校验流程，请在版本强制更新后设置为空）

    {pass_platform,[]}, %不需要校验的平台号例如{pass_platform,["iphone","ipad"]}

    {error,"ewp_verify_fail"} %校验失败的返回模板

    ]}

2 文件上传流程

2.1 通过内管

客户端将需要上的文件夹（例如iphone的app.app），根据client_file/Platfrom/Version/file这样的路径结构放好。 例如client_file/iphone/1.1.1/APP.app，client_file/android/1.1.1/apk等，然后打包成client_file.zip（包里不包含client_file文件夹）, 通过内管的资源管理，校验文件上传，上传到服务器。

2.2 通过脚本

客户端文件结构跟内管上传的结构一样，但不需要压缩，将client_file文件夹放到ewp有读写权限的目录下，然后执行脚本。 例如 client_file文件放在`/var/www/apps/tmp下，执行脚本如下： /use/local/bin/upload_verify_file /var/www/apps/tmp/client_file 脚本原理调用ewp里面的校验文件更新接口。

文件上传都会调用 sec_tamper_verify:update_file/1接口,参数是校验文件所在的目录（自定义），后台会遍历整个目录，根据目录的结构得出版本与平台， 再根据约定从客户端包中读出所需的文件，算出content值,计入mnesia数据库。算出content值后，服务器会将读出的文件摘要（content值）重新写到文件名为 file_content的文件中，并且将文件存放到校验文件的目录中（在security.conf中的client_verify_file字段所指定的目录).

文件结构为client_verify_file/Platfrom/Version/file_content. 　

注意：

上传文件时，文件夹命名，文件结构一定要按照上述约定 脚本上传文件结构与命名：client_file/Platfrom/Version/file.或 client_file/Platfrom/Version/Target/file

如果出现同一平台``同一版本下对应不同的手机系统版本有几个客户端包，就在version的下一级目录添加一级目录target

拿邮储项目为例：现在有3个包一个是iphone 1.5.0 的包，另外两个是安卓 1.5.0 对应不同系统版本的包，android_lephone2.0 android2.0.

所以这三个包分别的路径是

client_file/iphone/1.5.0/PSBC.app client_file/android/1.5.0/android_lephone2.0/bankps.apk client_file/android/1.5.0/android2.0/bankps.apk

其中Target字段名以user_agent中的target为准

平台 Platfrom file 文件类型
iphone iphone APP.app 文件夹
android android android.apk apk文件
ipad ipad Ipad.app 文件夹
androidpad apad apad.apk apk文件

file字段的命名可根据各个项目的项目名而定，但后缀跟文件类型要一样，ios平台的file 都是后缀为.app 的文件夹。android平台则为.apk文件

凡是校验文件的上传都要加经过脚本或内管上传，上传成功后会生成记录文件摘要的文件，以作备份之用。

3 信道改造

为了兼容已发布的客户端，所以信道版本会提升到1.3，在调用防篡改校验模块时，会比对客户端使用的信道版本号，根据配置文件的配置， 会判断是否走防篡改流程。
