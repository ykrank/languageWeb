产品文档

客户端开发规范

1 需求规格说明书

针对功能需求,形成需求规格说明书.说明书中需要详细说明此功能的原因,目的,功能点.

2 详细设计文档

需求文档形成后,需要对功能做详细的设计,形成详细设计文档.
详细设计中基本包含了实现中可能遇到的所有内容.如流程图,各个功能的实现方案,扩展方案,技术难点的解决方案等.

3 测试用例

根据需求及设计文档中的内容,建立测试用例.测试用例的维护是个长期的工作.
初期的测试用例需要包含所有的功能点的测试.协调好各种测试资源及环境搭建.
测试用例包含两方面:客户端native接口测试,功能单元测试.
测试用例的目的:

  • 保证各个功能点及逻辑的正确行
  • 使研发人员了解所做的工作的目的.
  • 防止后期开发进入新的问题.

测试用例的要求:

  • 能够覆盖所有的功能
  • 能够覆盖所有的分支场景.
  • 能否覆盖90%以上的代码.

    4 任务计划表

    根据需求及详细设计文档安排开发计划.
    开发计划需要对功能进行分解,将大的功能点分解为细化的小的功能点,并评估这些功能点的工作量.形成计划表.功能点的分解以不超过三天为益.

    5 建立ticket列表

    根据计划中的功能点,建立相应的ticket任务单.通过任务单我们可以方便的检查是否有遗漏的工作和查看剩余的工作内容.并在后期开发过程中将代码与ticket关联.

    6 开发工作

    按照详细设计及任务计划表中的顺序安排开发工作. 对于新功能的开发需要在分支上进行.防止由于未开发完成的功能对主干代码的污染.
  • 开发过程中要考虑API接口的提供方案
  • 在开发过程中发现的逻辑问题需要补充相应的native测试用例和功能测试用例.
  • 每个小的功能开发结束后,要主动要求代码评审.目前的代码评审建议通过phabricator进行.
  • 代码注视,需要养成代码注视的习惯.对于复杂的算法和特殊的处理需要详细的注视说明.代码注视量要达到80%.

    7 支持文档

    功能开发结束后,需要为用户提供相关的API文档,使用说明文档.为后期的代码维护人员提供架构设计文档.
  • API文档除了介绍接口的功能外,对于复杂的接口需要提供使用实例.
  • 使用说明文档需要提供用户关心的内容,比如如何开启,关闭功能.要实现某一个功能要如何实现,不同的效果要如何配置等等.

8 测试阶段

测试过程中的问题以ticket形式体现.ticket中包含问题的描述,重现的步骤. 对于开发人员在处理问题过程中,需要形成问题的原因,解决方案,以及后期测试的方案等相关内容.方便以后问题的总结,分析及考核.
