session概要设计

  • 现状
  • 问题
    • 会话保持
    • 数据一致性
  • 解决方案

1 现状

在EWP5.1中，我们加入了Session持久化功能，目的是使session超时后可以将会话中部分数据存储在持久化存储中，以便下次该会话被访问时依然可以读取到这些数据。

2 问题

2.1 会话保持

在实际项目中，会话保持会受f5分发策略影响。用户如果使用3G网络访问，其ip地址很可能会发生改变，当f5使用ip地址进行分发时就会出现将当前会话的请求分发到其他服务器的情况。用户便会看到会话超时的提示。

2.2 数据一致性

Session持久化功能并不能解决上述问题，并且如果持久化功能应用于上述场景中，可能会出现会话数据不一致的问题。原因是分发出错的请求会读取持久化的数据，而该会话很可能在之前的操作中已经修改了部分会话数据，因为Session并未超时，所以这部分修改并不会同步到持久化存储中。此时便会出现几个服务器之间读取同一会话的数据不一致。

3 解决方案

我们需要在活跃session和持久化session之间增加一个session共享机制，保证在当前服务器未读取到session数据时优先到保存该活跃session的节点上读取数据。因此，我们需要增加以下功能点：

  1. 修改SessionId生成流程，在生成的id后拼接上当前节点名称的哈希值，表明该Session存储在当前节点。
  2. EWP启动时，在所有服务器之间共享一个ram_copies mnesia表，并将节点哈希值和节点名称的映射关系存储在表中，哈希值作为Key。
  3. 在当前节点无法根据SessionId获取会话数据时，解析生成该会话的节点的哈希值，并在mnesia表中查找到对应的节点，并使用RPC到指定节点读取会话数据。
  4. 以上功能需将各服务器按集群部署，以保证RPC的正常工作。
