2 设计目的

在以往的测试过程中，存在着一些问题，影响了测试的效率和质量。一来有不少测试用例涉及到程序的最基本功能接口，如数据存储、加解密等，这些接口的功能都比较独立，输入和输出都很简单、唯一且稳定，但却造成测试人员的大量重复工作，耗费了时间。二来有关api接口的兼容性测试，在以往的测试用例中无法很好地被覆盖到，测试人员也不可能关注到所有api接口的变更情况。

针对上述问题，我们需要设计实现一套native的单元测试方案，将底层的功能接口测试交由程序去执行，将测试人员从繁复而底层的测试过程中解放出来，更多地去关注集成测试的效果。

设计的核心方案是:

  1. 对于输入输出唯一且稳定的功能接口，尽可能的细分到方法级别，通过给定固定输入，判断输出是否正确的方式，来确定接口功能是否正常。可以使用JUnit单元测试框架构建测试流程，进行每个接口的测试，通过断言来判断测试结果，并将最终结果输出到控制台或本地文件中。
  2. 针对lua接口和api接口，通过固定参数类型直接调用方法的方式，来测试接口的兼容性。测试结果可以通过打印标识位及是否编译报错来确认。

3 Lua接口功能测试

3.1 功能测试

在现在产品提供的Lua接口中，有以下接口可以通过native测试方案测试，基础测试方法为编写脚本直接调用的方式来测试这些接口的功能是否正常。根据接口功能的不同可分为不同的测试方案。

注：为提高测试的准确性，每个接口都需要多准备几组不同的测试数据。

  • document:getElementsByProperty()
  • document:getElementsByName()
  • document:getElementsByTagName()
  • document:getAllEventElements()
  • document:getEventElementsByListener()
  • document:getEventElementsByName()
  • utility:trim()
  • utility:length()

该组接口功能都较为独立，可通过固定输入查看输出是否正确的方式测试。以下测试方法基于JUnit测试框架，比如utility:length()接口：

public void test_utility_length() {
    String s = "1234567890";
    String testLua = "local len = utility:length(" + s + ");";
    testLua += "nativeTest:save(len);"
    DemoLua.mResult = null;
    mLuaControl.doString(testLua);
    // 将结果和10做比较，若不一致，则有问题
    assertNotNull(DemoLua.mResult);
    assertTrue(s.length().equals(DemoLua.mResult.toString()));
}

其中nativeTest:save(len)是一个扩展Lua方法，会将传入的参数转为对应的java数据类型，保存到DemoLua.mResult中。后续通过DemoLua.mResult来做结果验证。

  • database:addData()
  • database:getData()
  • database:deleteData()
  • kv:put()
  • kv:get()
  • kv:del()
  • utility:base64Encode()
  • utility:base64Decode()
  • file:write()
  • file:read()
  • file:remove()
  • file:isExist()

该组接口功能大都涉及到数据读写，无法通过固定输入查看输出的方式测试，但可通过组合调用的方式测试。如database的相关接口：

public void test_database() {
    String key = "key1";
    String value = "test";

    String testAddData = "local res = database:addData(" + key + ", " + value + ");";
    testAddData += "nativeTest:save(res);"
    DemoLua.mResult = null;
    mLuaControl.doString(testAddData);
    // 将结果和0做比较，若不一致，则保存失败
    assertNotNull(DemoLua.mResult);
    assertTrue("0".equals(DemoLua.mResult.toString()));

    String testgetData = "local res = database:getData(" + key + ")";
    testgetData += "nativeTest:save(res);"
    DemoLua.mResult = null;
    mLuaControl.doString(testgetData);
    // 将结果和mValue做比较，若不一致，则addData和getData有一个有问题
    assertNotNull(DemoLua.mResult);
    assertTrue(value.equals(DemoLua.mResult.toString()));

    String testdeleteData = "local res = database:deleteData(" + key + ")";
    testdeleteData += "nativeTest:save(res);"
    DemoLua.mResult = null;
    mLuaControl.doString(testdeleteData);
    // 将结果和0做比较，若不一致，则删除失败
    assertNotNull(DemoLua.mResult);
    assertTrue("0".equals(DemoLua.mResult.toString()));

    DemoLua.mResult = null;
    mLuaControl.doString(testgetData);
    // 将结果和null做比较，若不一致，则deleteData和getData有一个有问题
    assertNull(DemoLua.mResult);
}
  • utility:escapeURI()
  • utility:unescapeURI()
  • json:objectFromJSON()
  • json:jsonFromObject()

该组接口既可以用固定输入查看输出的方式测试，也可通过组合调用的方式测试。如json的相关接口：

public void test_json() {
    String content ="{\"one\":1, \"two\":2, \"primes\":[2,3,5,7]}";

    String testLua = "local objects = json:objectFromJSON(" + content + ")";
    testLua += "nativeTest:save(res);"
    DemoLua.mResult = null;
    mLuaControl.doString(testLua);
    // 测试objects的类型
    assertTrue(DemoLua.mResult instanceof CLEntity);

    CLEntity clEntity = (CLEntity) DemoLua.mResult;
    assertTrue("&one=1&two=2&primes=[2,3,5,7]".equals(clEntity.toString()));

    -- 反向转换，用组合方式测试
    testLua = "local objects = json:objectFromJSON(" + content + ")";
    testLua += "local res1 = json:jsonFromObject(objects);";
    testLua += "nativeTest:save(res);"
    DemoLua.mResult = null;
    mLuaControl.doString(testLua);
    -- 测试res的值
    assertTrue(content.equals(DemoLua.mResult.toString()));

    -- 也可以通过固定输入的方式
    testLua = "local objects = {a=15, b=22, zz={1,3,t=5}}";
    testLua += "local res1 = json:jsonFromObject(objects);";
    testLua += "nativeTest:save(res);"
    DemoLua.mResult = null;
    mLuaControl.doString(testLua);
    -- 测试res的值
    assertTrue(content.equals(DemoLua.mResult.toString()));
}

3.2 兼容性测试

lua的兼容性测试不需要考虑功能，因此理论上所有的lua接口都可以通过native方式测试。

由于android现在的lua接口都是通过jni的方式实现，主要实现都放在java层，jni在调用java方法时，本身就有严格的参数类型限制。因此测试方案为在脚本调用lua接口之前，记录一个标记位，然后在lua接口的java实现中，同样记录一个标志位即可。例如window:alert()的兼容性测试：

nativeTest:log("window:alert()", "", "start");
window:alert(1);

然后在java层LuaWindow的alert(Object[], int, final int)方法中记录

log("window:alert()", "", "end");

即可。

注：对于多可选参数的lua接口，需要准备不同参数的多组测试数据。

4 布局算法测试

在样式固定的情况下，针对同一款手机，报文中每个控件根据布局算法得到的位置信息肯定是唯一的。因此虽然native测试无法直观的看到布局的最终效果，但可以通过固定输入查看输出的方式来测试布局算法是否被修改。

测试方法为准备固定样式的页面报文，在每个控件布局结束后，都通过

log(widgetName, widgetCss, widgetAxis);

记录。其中widgetName为控件name属性，便于定位，widgetCss为控件样式字符串，widgetAxies为控件布局结果，包括宽高位置等。

注：为保证测试的准确性和覆盖率，需要准备不同样式不同控件组合的多组测试报文。

5 api接口功能测试

注：api接口测试需要使用混淆后的包。

5.1 兼容性测试

同lua的兼容性测试一样，api接口的兼容性测试不需要考虑接口功能。

那么只需要在本地代码中将所有api固定参数显式调用一遍即可，如果接口名称或参数做了修改，会报编译错误。运行时不必执行这些代码。

5.2 功能测试

功能独立，并且输入输出唯一且固定的api接口，可以进行native测试。测试方法为在本地代码中固定参数后，直接调用api接口，查看最终的结果。可以使用JUint框架判断测试结果。例如com.rytong.emp.data.tool.Utils:isEmpty(String str)接口的测试：

public void test_utils_isEmpty() {
    String str = "this is test String";
    boolean res = Utils.isEmpty(str);
    assertFalse(res);
}

同lua功能测试一样，针对不同的接口，可分为固定输入查看输出和组合调用两种测试方式。

需要功能测试的接口包括:

  • com.rytong.emp.data：数据处理相关接口，可用组合调用方式测试；
  • com.rytong.emp.dom：dom操作相关接口，可用固定输入查看输出方式测试；
  • com.rytong.emp.net:网络操作相关接口，可用固定输入查看输出方式测试；
  • com.rytong.emp.security：安全相关接口，可用固定输入查看输出方式或组合调用方式测试；
  • com.rytong.emp.tool：工具接口，可用固定输入查看输出方式测试。

注：为提高测试的准确性，每个接口都需要多准备几组不同的测试数据。
