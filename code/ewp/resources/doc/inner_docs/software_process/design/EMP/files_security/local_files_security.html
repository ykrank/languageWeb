客户端本地文件包括打包资源文件,离线资源下载的资源文件,本地数据库存储的数据. 比较了各个加密算法的性能,AES是性能最好的.因此我们选择AES作为我们的文件加密算法.

加密算法速度比较.rtf
加密算法性能测试报告.pdf

1 预置资源

预置资源包括图片,xml文件等.这里文件采用AES加密. 由于预置资源的加密需要在打包时处理,因此只能使用相同的秘钥. 产品需要提供加密预置资源的工具.由资源管理后台提供加密后的预置资源包.

这里说明一下资源管理后台对预置资源加密的方案:

  1. 加密秘钥需要有用户指定,可以通过输入的方式.此密钥需要预置到客户端,用于资源的解密操作.
  2. 加密方式可以选择AES,SM4或者不加密.甚至可以让用户自定义加密算法.客户端的解密算法要与之向对应.
  3. 对单个文件进行加密,如果是zip,需要将zip包解压然后对zip中的每一个文件进行加密.

2 离线资源

离线资源使用<<本地数据存储安全方案.rtf>>文档中关于无账号密码的加密方案. 由设备IMEI号、设备标示、预置三个字符串密钥（三个string保存在不同的原代码文件中）组成原始message message = IMEI + deviceID + string1 + string2 + string3; 使用sha1、md5算法计算message的hash值得到36字节(sha1为20字节，md5为16字节)密钥串即： secret = sha1(message) + md5(message) key：从secret中顺序取前32字节 iv：从secret中倒序取后16字节 对于无法取得IMEI号的设备，使用空字符串取代，对于无法取得deviceID的，同样使用空字符串取代（也可以使用其它设备信息取代，如j2me可使用microedition.platform得到手机的品牌和型号，Nokia手机的返回值格式为“Nokia6310i/4.42，iphone可通过[UIDevice currentDevice]取得设备信息等等）。

数据库加密采用与离线资源相同的方案.但是秘钥不能相同.

5 如何满足多变的项目需求

为了让项目能更好的满足客户端需求, 我们需要提供加密的扩展功能.通过重写或代理的方式插入自定义加解密函数,项目中可以定制自己的加密算法.在我们存取文件时,主动调用这些接口.

考虑到不同的资源可能会有不同的加密方式,我们需要将文件的类型(预置资源,数据库,离线下载,lua写入的资源等类型)作为接口的参数.根据不同的文件类型选择相应的加解密方案.

比如:

  • (NSData )encryptData:(NSData )originData type:(int)type; 加密数据
  • (NSData )decryptData:(NSData )data type:(int)type;

6 国密

由于在5.2中产品已经支持了国密的算法.因此除了前面描述的AES加密外,还要支持国密算法对资源文件的加解密. 通过项目配置选择相应的加解密算法.
同样需要提供国密算法的加密工具.以便于对预置资源的加密.
