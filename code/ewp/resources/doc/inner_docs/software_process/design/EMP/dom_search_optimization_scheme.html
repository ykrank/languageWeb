2 引言

目前产品代码中document:getElementsByPerporty()等接口，是通过遍历整个DOM树，比对属性值的方式实现的。可以预见，在DOM树结构庞大或者接口调用次数较多时，该方式执行效率会比较低下。而通过缓存查找数据的方式，可以有效减少节点查找的时间消耗。我们需要测试遍历节点方式的具体时间消耗，同时测试通过索引表建立缓存方式查找节点的执行效率。最终比较这两种测试结果，找出优化查找DOM节点效率的方法。

本测试基于Android 5.3.17版代码，测试接口为getElementsByClassName()。

4 遍历查找方式

4.1 测试结果

测试结果为5次测试的均值，单位为ms。

直接执行测试报文：

N=? 100 500 1000
报文A 18 70 152
报文B 14 56 104
报文C 12 38 86
报文D 154
报文E 120

将报文A内的getElementsByClassName接口调用修改为10次，测试数据：

N=? 100 500 1000
报文A 156 828 1552

4.2 结果分析

  1. 页面内节点数量越多，遍历查找所需的时间越长；
  2. 页面内节点数量一定时，遍历的效率受页面内含有class属性的节点总个数影响。总个数减少，查找效率提高。分析是代码内对于没有class属性的节点，直接跳出，省去了取值比较的过程；
  3. 节点的嵌套层级对遍历效率的影响不大；
  4. 重复执行查找接口，时间花费会成倍数增长。

5 索引表方式

为减少节点查找的时间消耗，我们可以通过索引表方式缓存待查找的节点数据。

5.1 实现方法

以getElementsByClassName()接口为例，在解析报文时，将含有class属性的节点，以class属性值拆分后的样式名（后续简称class属性）为key，节点队列为value的map形式保存在客户端缓存中。执行getElementsByClassName()接口时，直接在缓存map中获取对应class属性key的节点队列，并返回。

5.2 测试结果

测试结果为5次测试的均值，单位为ms。

直接执行测试报文：

N=? 1000
报文A 48
报文D 28
报文E 22

将报文A内的getElementsByClassName()接口调用修改为10次，测试数据：

N=? 1000
报文A 452

将报文A内的getElementsByClassName()参数改成不存在的样式，如aaa，测试数据：

N=? 1000
报文A 0

5.3 结果分析

  1. 该方法比遍历方法效率提高明显，因为该方法省去了遍历查找的过程，直接在缓存中取值；
  2. 当查找未命中时，不需要遍历节点，也不需要拼接结果，时间花费几乎为0；
  3. 页面内含有目标class属性的节点越多，获取时间越长。这是由于目前缓存map中保存的value是节点队列，而接口的返回值是数组，返回时需要循环队列，拼装结果数组。那么实现效率就会受到队列中的节点个数影响。

注： 通过索引表方式实现时，并没有考虑及详细记录解析阶段加入索引表处理对解析效率的影响。不过从测试过程来看，影响并不是很明显。

6 最终优化方案说明

根据以上测试数据，可以确定优化方案如下：

  1. 采用索引表方式。
  2. 目前产品提供getElementById、getElementsByName、getElementsByTagName、getElementsByClassName、getElementsByProperty五个接口，考虑到接口使用的便利性和频率，客户端需要保存以id、样式名（classname）为key的两个缓存队列。
  3. 在页面解析时，以key-value的map方式，将待查找的节点保存在客户端缓存中。其中key为待查找的属性（即id或样式名（classname）），value为包含对应属性的节点队列。
  4. 页面跳转时，要清空缓存数据。
  5. 由于id属性的唯一性，开发模式下，调用getElementById接口时，若获取的结果大于1个，需要提示错误。

6.1 DOM树变更时的处理

在页面DOM树更新时（即调用setInnerHTML、appendChild、insertBefore、removeChild接口后），为保证数据准确性，需要对缓存数据做处理。

setInnerHTML的处理:

  • 遍历被替换的旧节点及其子节点，如果节点含有id或者class属性，则将缓存中该节点的数据清除；
  • 解析新节点报文时，按照id或者class属性更新对应索引表数据。

appendChild、insertBefore`的处理:

  • 先判断调用的控件是否在dom树中，没有则不做额外处理；如果在，则遍历新子节点及其子节点，判断是否含有id或者class属性，添加缓存。

removeChild`的处理：

  • 先判断调用的控件是否在dom树中，没有则不做额外处理；如果在，则遍历新子节点及其子节点，判断是否含有id或者class属性，清除缓存。

注：判断控件是否在dom树中的方法，依平台不空可能系统提供的接口也不一样，这里不做统一，只要能实现功能即可。

6.1.1 确保结果集顺序正确

调用getElementsByClassName获取节点时，结果数据通常不是单一的，我们需要保证获取的结果集顺序正确。但在DOM树变更后，缓存里的节点顺序很可能不准确。如果采用遍历DOM树，重新生成缓存的方法，虽然能保证节点顺序，但效率较低。这里我们可以采用另一种方法处理。

首先在页面解析时，为每个element节点赋值一个层级索引。可采用如下数据结构表示：

ElementIndex {
    ElementIndex mParentIndex;
    int mIndex;

    String toString() {
        if (mParentIndex != null) {
            return mParentIndex.toString() + "_" + String.valueOf(mIndex);
        }
        return String.valueOf(mIndex);
    }
}

这样body的层级索引toString()结果是1，body子节点的层级索引toString()依次为1_1、1_2...二级子节点的层级索引toString()依次为1_1_1、1_1_2...1_2_1、1_2_2...

如此缓存的时候不需要考虑顺序，只需要在获取拼接结果集的时候依据层级索引排下序即可。

setInnerHTML的处理:

  • 同页面解析，在解析新节点报文时，按照遍历顺序给每个新节点赋值层级索引。

appendChild`的处理:

  • 先判断调用的父节点是否在dom树中，没有则不做额外处理；如果在，执行下一步；
  • 获得新节点同级子节点中，最后一个有效子节点的层级索引，将其mIndex值加1，作为新节点的层级索引mIndex值；
  • 遍历新节点的子节点，赋值层级索引。

insertBefore的处理:

  • 先判断调用的父节点是否在dom树中，没有则不做额外处理；如果在，执行下一步；
  • 获取被插入子节点的层级索引，将其mIndex赋值给新子节点；
  • 向后遍历同级子节点，每个子节点的索引mIndex加一；
  • 遍历新节点的子节点，赋值层级索引。

removeChild`的处理：

  • 不需要处理，直接删除。

注：判断控件是否在dom树中的方法，依平台不空可能系统提供的接口也不一样，这里不做统一，只要能实现功能即可。

6.1.2 说明

  1. 经测试发现，采用层级索引方案处理DOM树变更的情况时，由于存在结果集排序的环节，如果结果集数量在50个左右，则获取速率和遍历DOM树重新生成缓存相差不大；若结果集数量大于50个，则遍历DOM树方式效率更高；若结果及数量小于50个，则层级索引方式效率更高。考虑到在实际应用过程中，一般而言获取的结果都远小于50个，因此我们依旧采用层级索引方式。
  2. 为进一步提高效率，可以设置一标志位，只有DOM树发生变更时，标志位开启，此时处理结果集需要排序；同时将排序后的结果更新到查找缓存中。然后关闭标志位，后续在获取时就不需要排序了。
  3. 排序工作只适用于getElementsByClassName接口。id属性具有唯一性，因此getElementById接口的返回结果只能是一个。
