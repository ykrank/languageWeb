2 引言

2.1 需求概述

　　华夏手机银行系统存在抢购功能，短时间大量用户集中使用，致使整个系统响应用户能力下降，导致使用其他功能的用户不能正常使用手机银行。
　　为解决短时间内大量用户的集中使用而不影响用户的体验，采用分布式部署服务，为保证信息分发的相对安全，客户端在原有信道基础上需要支持多https信道功能。 　　

2.2 参考资料

《华夏手机银行分布式改造方案 1.0.1》

3 设计说明

3.0.1 服务端功能需求

  • 服务端需要设计一个基于EMP信道的同步域名列表的请求，在EMP信道建立成功后客户端通过该请求获得如下格式的内容，格式如下：

    • 同步列表返回格式：1字节的type + 4字节的长度 + 域名列表
      当服务器的域名列表为空时，域名列表内容为 “{}” 即可

    • 域名列表内容格式 json 字符串

          {
      "mode": "https",
        "url": [
            "http（或者https）://host:port",
            ...
        ]
    }
  • 服务端在解析EMP握手请求时，当请求的head中有X-Https-List字段时，需要在响应的head中加入X-Https-List字段，内容为服务器域名列表的最新版本号，此版本号由服务器维护，最好为一个小数点格式（如：1.0，1.1，……）。

3.0.2 同步域名列表请求、域名列表保存方案

　　客户端接收到服务器返回的同步列表内容时，根据格式解析出域名列表的数据，并将域名列表加载到内存中以便后面使用，然后客户端异步方式，并且对域名列表数据做一次MD5的计算，并将该校验值和数据一起存在本地的文件中，版本号在服务器的响应的head中，以"X-Https-List"字段返回。文件保存格式如下：

　　　　2个字节版本号 + MD5校验值（16位）+ 域名列表内容 　　

3.0.3 域名列表文件读取方案

读取存在本地的域名列表文件，解析并用MD5校验值校验其完整性

  • 文件完整，域名列表解析正确 --- 读取域名列表到内存中。
  • 文件不完整 -- 删除文件，重新请求列表同步，请求返回走域名列表保存方案。
  • 无文件 -- 重新请求列表同步，请求返回走域名列表保存方案。

3.0.4 客户端实现逻辑

1、客户端需要支持通过配置来开启、关闭多信道支持的功能。该功能最好以库的方式提供，不添加该库时不影响原有的其他功能。

2、客户端配置开启功能后，在建立信道前，加载域名列表文件，解析出列表文件的版本号，

3、建立信道时，在握手的请求中，添加X-Https-List字段，内容为域名列表文件的版本号，当文件不存在时可默认为"0.0"。

4、服务器在EMP信道握手成功后，在响应的head中也返回X-Https-List字段，该字段为服务器域名列表文件最新的版本号(当客户端不发送这个字段内容时，服务器不需要返回)。

5、客户端将两次版本号做比较，当本地版本号不等于服务器版本号时，客户端发起同步域名列表请求，请求返回时，按域名列表保存方案执行。当版本号相等时，客户端执行域名列表文件读取方案。

6、解析域名列表内容并且校验其完整性，如果内容完整，并且https域名列表不为空，保存域名列表到内存中，开启后续的多信道功能：后续的每次请求的请求头中加入"X-Emp-Cookie"字段，内容为session id，字段的内容为最后一次请求成功后，响应的head中"X-Emp-Cookie"字段的对应值。

7、当每次请求开始前，获取请求的"http（或者https）://host:port"，通过与内存中的域名列表做比对，存在该域名时可以继续访问。当检测到域名列表之外的请求时，为了确保和服务器的同步，客户端需要做一次同步域名列表的请求。和响应中解析出域名列表再次做比，仍然不存在时，此时的客户端可能已经收到攻击，处理逻辑：

  • 终止请求，弹框提示"域名不存在"，内容要可配置；
  • 是否关闭客户端：不需要关闭
