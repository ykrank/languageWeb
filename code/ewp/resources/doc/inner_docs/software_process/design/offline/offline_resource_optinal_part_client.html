可选资源增量更新 - 客户端设计

0.1 背景

可选资源均为插件包，当用户需要更新的资源较大或网络条件不好的情况下，更新整个可选插件包速度很慢，甚至很可能失败，用户体验较差。我们需要增加可选插件包中部分资源更新方法，使得用户可以每次更新较小的资源。

1 需求

  1. 增量更新可选插件包的功能
  2. 添加可选资源更新时进度条功能

2 对应lua 方法

1.offline:downOptionalFile(filename, parameter)；

  • Description:

    根据插件包名，比较客户端已下载与服务器返回的描述，记录需要删除及下载的文件。
    需要兼容之前功能。

  • Parameters:

    1. filename(required) 文件名（依.zip结尾）

    2. parameter(optional)

      • 类型：Table

      • 格式：{processCallback=processCallback, finishedCallback=finishedCallback,appname = "emas", info = "message"}

        • processCallback(infotable, down, total) 下载进度回调方法
          参数说明： 

           * infotable_(required)_ Table 类型，{appname="当前应用名"，filename="文件名"}
 * down_(required)_    number类型 为当前已下载的 
 * total_(required)_ number类型 为总共需要下载的<br/>

          a) 若是整个zip包更新,根据已下载的文件大小做回调。参考下载进度回调

          b) 若是部分资源更新。每下载一个文件执行回调。
          c) 文件的删除不计入下载进度
          d) 执行顺序：先删除再依次下载资源

        • finishedCallback(infotable, failtable) 下载结束回调方法
          参数说明：

           * infotable_(required)_ table型；{appname="当前应用名"，filename="文件名"}。
 * failtable_(optional)_ table型；key值为默认值(1，2，3...)，value:若整个插件包更新失败，则为插件包名；否则为更新失败的文件名。
     示例：  整个插件包失败：{1："test.zip"} ；  部分资源失败：{1："test/css/base.css",2:"test/xhtml/main.html"}
          • appname
          • info

          注意：

          function finishedCallback(failtable) end

          local params = {processCallback=processCallback, finishedCallback=finishedCallback,appname = "emas", info = "message"}

          offline:downOptionalFile("zip1.zip", finishedCallback, params)；

    此种情况下客户端只会对第三个参数params中 finishedCallback 对应的回调方法做处理。

  • Return:
    无

  • Examples:

      -- 下载进度回掉：
  function processCallback(infotable，downNum, totalNum)

  end

  -- 下载结束回掉：
  function finishedCallback(infotable，failtable)
      if failtable and #failtable > 0 then
          window:alert("资源更新结束，但有部分资源未更新成功。");
      else
          window:alert("资源更新成功。");
      end
  end

  local params = {processCallback=processCallback, finishedCallback=finishedCallback,appname = "emas", info = "message"}；

  offline:downOptionalFile("zip1.zip",params);

3 平台差异

3.1 下载进度回调

3.1.1 Http下载方式：

AD平台： 每下载10%执行回调
IOS平台：每接收到一个数据包执行回调

3.1.2 Tcp下载方式：

AD平台： 每下载10%执行回调
IOS平台：每接收到一个数据包执行回调

3.1.3 WebSocket下载方式：

各平台统一：因为该方式为异步下载，一次性返回所有数据所以进度条下载完直接做100%回调

3.2 删除进度回调

各平台统一：删除完资源删除完做100%回调

3.3 重复下载进度回调

各平台统一：只做一次100%进度回调，不下载数据
