客户端离线资源WebSocket下载支持

1 1 需求

参照离线资源下载协议，离线资源增加使用WebSocket下载方法，实现客户端离线资源的安全下载功能。

1. 客户端能根据接解析服务端在描述文件中传输方式的字段，根据该字段标识的传输方式下载资源。客户端根据优先级"webSocket" > "tcp" > "http"判断连接和下载方式。
2. 客户端webSocket下载支持的传输方式：明文传输、EMP信道明文传输、EMP信道密文传输。
3. webSocket下载需要支持同步和异步两种方式。
4. webSocket数据传输协议优化。

2 2 处理流程

2.0.1 2.1 离线下载方式选择流程

  1. 客户端通过请求ota/resource_update或者通过信道获得下载描述文件.
  2. 在请求成功的返回数据中，新增了一个"w"字段:WebSockethost，记录连接WebSocket的host。客户端获取该字段的值，当不为空时通过Websockethost开始WebSocket下载。
  3. 当"w"字段为空时，再去获取"t"字段的值，它记录了tcp连接下载的port，当port不为空时开始tcp下载，为空时选择http方式下载。

2.0.2 2.2 WebSocket协议

webstocket是html5的一种新的协议，它实现了浏览器与服务器的双向通道，使得数据可以快速的双向传播.

通过一次简单的握手，建立了客户端和服务器的联系后，服务器可以主动推送信息给客户端，而不需要客户端的反复请求。

一个websocket连接是客户端与服务器端在http协议的初始握手阶段将其升级到websocket协议来建立的，其底层还是tcp/ip连接。

2.0.2.1 优点：

  1. 客户端和服务器端之间数据交流的表头比较小。大概2个字节

  2. 服务器和客户端可以主动的发送数据给对方

  3. 不需要有频率的创建TCP请求和销毁，节约宽带和服务器的资源

2.0.3 2.3 WebSocket下载流程

  1. 确定下载方式。当资源描述中含有'w'字段时,说明需要使用websocket下载方式进行下载.

  2. 建立与服务器的连接。WebSocket初始化时，会根据“w”字段提供的Websocket host和服务端建立安全连接。连接失败后，直接用http连接下载。建立连接的接口为ota/ws_connect.Websocket host + ota/ws_connect 作为建立websocket连接地址.

  3. 处理上送数据。WebSocket连接成功后，对上送的数据进行处理。当"st"为3时，websocket暂定明文传输。
    连接建立成功后,向服务器发送EMP资源和H5资源的hash值.如果此时存在之前下载失败的文件,需要发送失败的文件一同上传给服务器.
    客户端请求上送的数据格式: 

     EMPTCPVersion(2 byte) + RequestMessages，
 其中RequestMessages由若干消息组成，每个消息格式为Type(1 byte) + Length(4 bytes) + Content(Length bytes)组成。  
 每个字段意义如下：  
 EMPTCPVersion第一字节表示大版本，第二字节表示小版本。  

 Type表示消息类型。
 0为session id，

 1为签名，

 2为序列号或者组别+序列号. 在信道1.3及之前版本,客户端只上送序列号,信道1.4及之后版本上送组别及序列号.

 3为文件路径或包含其他信息的json数据(我们叫它paramsJson)。所有传输方式都需要上送文件路径。 在信道传输中,Type 1 的签名即是对Type 3 内容的签名,信道密文传输方式,需要对type 3内容进行加密.

 Length表示消息的长度。  
 Content为消息内容。文件路径与原协议保持一致，使用JSON格式表示。 使用EMP信道时只对Content内容签名和加密。服务端在TCP下载响应中增加错误码3，表示安全验证失败。  
 使用EMP信道密文传输时，对文件数据的Content进行加密。使用EMP信道传输时，对序列号消息的Content+文件路径消息的Content（密文传输时为加密后的内容）进行签名。  
 paramsJson的数据格式:  

 {
 "hash":hash,
 "h5_hash":h5_hash,
 "platform":"iphone"/"android",
 "resolution":"实际分辨率",
 "from":appname,多app时.
 "downlist":ewp失败列表(array),可选
 "h5downlist":H5失败列表(array),可选
 "info":info,offline:update_resource(params)接口的params参数的info,可选
 }

  4. 处理下载数据.服务器接收到客户端发送的hash和文件列表后,将需要返回的所有数据一起返回给客户端.
    服务器响应格式为EMPTCPVersion(2 byte) + ResponseMessages，其中ResponseMessages由若干消息组成，格式与RequestMessages相同。 

    Type类型包括0、1、2、3、4、5.
0为资源内容，当使用信道传输内容时可能为信道明文,或者信道密文. 客户端是否对文件内容进行解密,依据为st字段中定义的类型. 

1为签名（使用EMP信道传输时会包括此消息），

2为错误信息，服务端程序异常时只会包含错误信息。Type为2时，Content表示错误代码：为0时表示未知错误，为1时表示无法读取指定资源文件，为2时表示无法解析客户端上送的路径，为3时表示安全验证失败。
列举几种错误的数据格式:
错误码为0时: 40(版本号) + 2(type) + length + 0(errorcode)  
错误码为1时: 40(版本号) + 2(type) + length + 1(errorcode) + 4(type) + length + fileInfo.

3为组别（1位byte） + 序列号（8位byte），格式为：Type(1 byte) + Length(4 bytes) + Content(Length bytes)。如：<<PackBin/binary, 3,0,0,0,9,1,0,0,0,0,0,0,0,1>> . 
服务器返回组别和序列号是在信道1.5及之后的版本.

4为文件信息,
{
"path":"resources/common/txt", 文件下载路径
"file":"aaaa_01.txt"， 文件名
"type":"ewp / h5",表示资源类型.
} 

返回文件数据时的数据格式: 40(版本号) + 0(type) + length + filecontent(文件内容) + 1(type) + length + hash + 3(type) + length + groudserial + 4(type) + length + fileInfo

5为结束标记,结束时的数据格式:40(版本号) + 5(type) + length + message(send over).
  1. 全部下载结束，当客户端接收到type类型为5时,表示服务器传输结束,客户端主动断开连接，清空缓存。当所有文件下载完成后，客户端须主动断开WebSocket连接，并且清空相关缓存。如下载过程中连接断开，客户端须根据已成功下载的文件更新本地资源描述，并重新请求客户端更新接口。

3 客户端本地保存策略

由于服务器一次性返回所有数据会导致客户端无法及时的处理,并且客户端本地的校验和数据保存会影响整个过程的性能. 因此我们需要在客户端接收到数据后,采用异步的方式处理这些数据.包括数据的解析,校验,文件描述的保存和入库,文件数据的持久化存储,下载列表的维护等.文件写入时与之前版本方案相同,需要先做文件完整性校验,在写入到硬盘中.

由于异步保存数据,需要建立缓存队列.在websocket加载过程中,如果出现网络中断或其他导致下载失败的情况,需要转为http下载. 需要注意的是,由于下载列表中的文件数据需要在文件处理成功后才能删除.因此我们要求在转为http下载之前, 需要先确保缓存队列中的数据已全部处理结束,以此来避免数据的重复下载.

4 失败列表的维护

websocket上送给服务器的失败列表为客户端合并后列表. 客户端对失败列表的维护方案参见离线更新策略 此失败列表的内容包括:

  1. 之前的更新中下载失败的资源.
  2. 在1列表的基础上去除本次更新中需要删除的资源
  3. 在2类别的基础上去除本次更新中需要更新的资源.
