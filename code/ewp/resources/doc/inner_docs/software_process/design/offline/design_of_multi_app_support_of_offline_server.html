离线资源多App支持服务器设计

    • 服务端

1 需求概述

随着用户增长和架构的需求，手机银行的后台服务将向微服务转变，由原来一个App后台提供服务转变为按业务划分的多个微服务后台提供手机银行App的服务。这样将会为手机银行App各业务提供更好的隔离性，降低各业务之间的影响。
为支持这种微服务架构，EMP离线协议也需要作出相应修改，以便各App的离线资源可以分开管理和下载。

2 功能需求

我们需要实现以下需求：

  • 新增离线协议版本，支持多App服务管理和更新离线资源，同时不影响旧版本协议功能。
  • 客户端支持维护多个App的离线资源。需将各App的资源描述分别维护。各App的资源存储路径同样要分开。
  • 客户端支持通过Lua脚本更新指定App的资源（必选和可选都支持）。
  • 服务端除了支持直接下载外还需支持转发下载请求。有可能存在通过一个EWP App集群下载其他EWP App集群或H5 App集群的资源
  • 转发下载请求时，服务器需提供可配置的缓存功能，以便提高资源的下载效率。

3 设计

为兼容旧版本信道协议，我们需要新增3.0版本协议来实现离线资源多App支持。

3.1 离线协议

3.1.1 资源来源标识

由于每个App分别维护各自的离线资源，因此我们需要在下载、resource_hash和resource_update接口的请求参数中增加from参数来标识访问哪个App的离线资源。

3.2 服务端

3.2.1 配置

增加配置：

{offline, [
    ......
    {redirect, [ %%配置需要转发请求的App
        {App, [
            {host, H},                %%向App转发请求时访问的主机地址
            {type, ewp | h5},         %%App服务类型。默认为ewp
            {protocol, http | tcp},   %%基于哪种协议进行下载转发，默认为http
            {handler, Mod},           %%指定转发处理模块，可自定义转发逻辑，
                                      %%不配置时使用EWP默认逻辑。
            {tcp_port, P},            %%使用tcp协议转发时使用的端口号
            {cache, boolean()|fixed}, %%缓存功能开关。缓存功能开启后，当资源
                                      %%文件第一次被访问时，会在本地内存缓存
                                      %%一份内容，如果有效期内没有被访问，则会
                                      %%清除缓存内容。
                                      %%如果配置为fixed，则有效期为固定值，即在
                                      %%有效期内，即使文件被访问，也不会更新有
                                      %%效期限。
                                      %%默认为false。即不开启缓存功能。
            {cache_expiry, Time},     %%缓存有效期，单位为秒。默认为5分钟。
            {max_connections, Max}    %%该App下载资源占用的最大连接数。
                                      %%默认为100。
        ]}
    ]}
]}.

3.2.2 H5离线资源管理

H5的离线资源需在EWP管理后台上传，上传方法为：

  1. 将所有资源保存在H5 App命名的文件夹下，如h5_app1/
  2. 将该文件夹压缩成zip包
  3. 将zip包在管理后台上传

上传后，客户端会同步相关资源。页面通过相对路径即可访问资源，例如上传时资源a.css在H5 App命名的文件夹下的相对路径时css/path/a.css, 那么在页面中访问css/path/a.css即可。

EWP需要将H5 App命名的文件夹保存在public/h5_resources目录下，以便为各H5 App分别保存资源。H5资源的描述文件保存在public/h5_resources/desc和public/h5_resources/sca_desc目录下，命名规则为“h5-From-common-Version.desc”。其中From为H5 App的名称，Version为离线资源的版本。

补充说明如下：

  • H5资源全部为普通、必选资源。
  • H5资源生成描述文件时，文件名称即为完整路径，因此不需保存p字段。
  • H5资源均默认为加密存储，确保更新。
  • H5 描述文件".desc"格式为：
{
    "hash": "c4f3cffb7d68e63531c60e74611a2d2f569173e7", // H5资源描述计算出的哈希值
    "h5": { //h5资源描述
        "path/file1" : { // 如: "demo/img/welcome.png"
            "p" : "h5",
            "mime" : MimeType,  // 如: "text/plain"、"image/png"...
            "r" : "c4f3cffb7d68e63531c60e74611a2d2f569173e7",
            "e" : 1,  //1,加密, 0,不加密 默认为0,且如果为默认,则省略此字段
            "u" : 1,  //1,确保更新, 0,非确保更新 默认为0,且如果为默认,则省略此字段
            "n" : "普通资源说明"
        },
        ...
    }
}
  • H5 App在resource_update接口中返回的描述文件更改为：
{
    "h":"http://127.0.0.1:4002/", //http下载地址
    "t":4060, //tcp下载端口
    "m":0, //资源更新提示方式。0为提示可选更新，1为提示强制更新，
                    //-1为不更新,

    "h5" : //h5资源更新列表
    {
        "s":"21468cbc6559727548342f5c6fa0915c01a6a1ea"//必选资源hash 服务端计算的描述文件哈希值，所有资源下载成功后，客户端将该哈希值保存，作为ota/resource_hash接口参数使用。

        "h5" : { //本地描述的资源路径（p字段），目前H5资源的p字段值均为h5
          "path/file1" :{ // 普通资源 如: "demo/img/welcome.png"
            "r" : "c4f3cffb7d68e63531c60e74611a2d2f569173e7",
            "mime" : MimeType,  // 如: "text/plain"、"image/png"...
            "u" : 1  //1,确保更新, 0,非确保更新 默认为0,且如果为默认,则省略此字段
          },

          ... //其他普通资源
        }
    }

    "h5_d" : //h5资源删除列表
    [
      "path/file5",
      "path/file6"
    ]
}
  • H5 App预置资源描述格式改为：
{

        "s" : "c4f3cffb7d68e63531c60e74611a2d2f569173e7"
        "path/file1" : 
        {
          "r" : "c4f3cffb7d68e63531c60e74611a2d2f569173e7",
          "mime" : MimeType,  // 如: "text/plain"、"image/png"...
          "e" : 1, //如果不需要加密存储,则没有此字段

3.2.3 上传H5资源信息接口

接口名称：resource/upload_h5_res
输入：zip包
输出：成功返回ok
失败返回Json {"error":错误信息}

3.2.4 离线资源转发

向EWP App转发请求方法：

  • 在请求中获取from参数，从而确认将请求转发到哪个App，并获取配置信息。
  • resource_hash和resource_update接口只需要向App请求描述文件，并在本地完成后续处理。如果开启缓存且已缓存对应的描述文件，则不转发请求。
  • 下载文件请求直接转发给App，如果开启缓存且已缓存指定文件，则不转发请求。

H5 App不需要转发。

3.2.5 获取描述文件接口

接口名称：ota/resource_desc
参数：platform 客户端平台
resolution 客户端分辨率，格式为320*480
version 离线协议版本
hash_type 摘要算法，值为SHA1或SM3
返回值：成功返回描述文件内容
失败返回Json {"error":错误信息}

3.2.6 文件缓存

  • 文件缓存功能使用ets + gen_server/gen_fsm process来实现
  • 每个App需对应一个process。
  • supervisor process来管理缓存process。
  • 根据配置决定是否启动对应App的缓存服务
  • 如果开启缓存服务，则需要在ets表中存储“{文件名,文件内容,时间戳}”格式的记录，以文件名为key值。
  • 定期检测ets每条记录的时间戳与当前时间的差值是否超过有效期，并删除这些记录。
  • 如果缓存配置为fixed类型，则不更新时间戳。否则每次文件被访问时，更新对应记录的时间戳。

文件缓存服务应提供一下API：
offline_file_cache:read(App, Path) -> undefined|binary()
获取文件缓存接口。如果没有缓存则返回undefined

offline_file_cache:clear(App) -> void()
清空缓存接口。

3.2.7 下载并发管理

  • 下载并发管理功能使用gen_server process来实现
  • 每个App需对应一个服务process
  • 每个下载请求都会对应一个process，我们需要在下载流程中调用gen_server:call向对应App的并发控制服务发送消息，通知并发控制服务当前下载process的id
  • 并发控制服务每收到请求时就与对应的下载process建立link，并且计数器加1。
  • 如果计数器达到上限，则返回错误信息，下载流程终止。
  • 由于建立了link，下载process退出时会向并发控制服务发送退出消息，此时并发控制服务将计数器减1。

下载并发管理服务应提供以下API：
offline_dl_ctrl:is_dl_busy(App) -> boolean()
判断是否下载繁忙接口。
