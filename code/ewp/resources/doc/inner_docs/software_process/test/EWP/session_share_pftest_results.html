1 测试内容

本次测试内容：Session共享优化后的性能。分别测试单节点、多节点情况下，Session操作的性能。

2 测试环境

2.1 测试机器

主、从节点两台服务器配置相同： 系统：CentOS release 6.5 (Final) CPU：2 × Pentium(R) Dual-Core CPU E5700 @ 3.00GHz 内存：2 G MySQL版本：5.1.73

2.2 测试程序

erlang版本：R15B (erts-5.9) 64-bit ewp版本：5.2 ebank版本：5.2 mysql版本：5.1.73

3 测试方法

本次测试是模拟ebank APP的一次检查手机帐号的请求，通过RPC在两个节点之间进行交互。所以，测试环境需要依赖ebank以及集群环境。 在ewp的session_service模块中，添加了三个测试用例，主要思路是在节点A注册session，在节点B用模拟的请求，通过performance_test:main/5性能测试函数不断的调用测试用例，使用cpu_usage:start/1统计函数统计时间，记录结果并分析。 测试用例：

  1. 单节点测试用例：pf_single_test(N_Pro, N_Times, LoopTime) 功能：测试单节点session操作效率，统计结果与多节点测试对比。 思路：在自节点注册session，并在自节点模拟ebank的检查请求，统计请求的平均时间和总时间。
  2. 多节点测试用例：pf_multi_test(N_Pro, N_Times, LoopTime) 功能：测试两节点间，通过RPC通信（请求节点空闲）的session操作效率。 思路：在远程节点B注册session，在自节点A模拟ebank的检查请求，保证每次请求都是通过RPC来操作session。统计请求的平均时间和总时间，与单节点测试统计结果对比，能够了解通过RPC方式操作session，对整体性能的影响。
  3. 多节点测试用例：pf_both_test(N_Pro, N_Times, LoopTime) 功能：测试两节点间，通过RPC通信（请求节点繁忙）的session操作效率。 思路：创建两个进程，一是通过RPC调用远程节点B调用单节点测试用例pf_single_test，使B节点CPU繁忙；二是在自节点A调用多节点测试用例pf_multi_test，模拟在远程节点繁忙的场景下，统计通过RPC操作session的效率。

参数解释：

  • N_Pro：并发数。
  • N_Times：执行测试函数的次数。
  • LoopTime：循环次数。 备注：测试函数的总执行次数为 LoopTime × N_Times。

5 测试总结

根据记录的统计结果，可以得出：

  1. 单节点的Session操作效率要高于多节点操作，范围在2-20倍。
  2. 在当前测试环境中，单节点操作并发量在100左右效率最高，多节点操作在并发500左右效率最高。
  3. 在调用节点繁忙时，会影响多节点操作的效率。
