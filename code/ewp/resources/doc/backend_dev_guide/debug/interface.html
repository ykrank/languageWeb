接口联调

本章节主要讲解如何后端接口联调

接口联调主要是做什么的？在开发期间，接口联调需要保证程序的输入、返回和接口文档一致。在联调测试阶段，联调需要在前面之上保证数据、业务逻辑的正确性。

本章节分两种情况讲述如何进行接口联调

1 1、模拟报文联调

  • 1.1 模拟后端输入输出（本操作建议使用simulator）

    • 1.1.0.1 第一步：配置模拟session配置make_session.conf 与公共参数配置public_params.conf，启动服务
    • 1.1.0.2 第二步：打开浏览器访问http://192.168.241.3:4005/admin，登陆后记录后台日志中sessionId，在后台运行的erlang shell中执行读取session如下：
      session_service:read_by_id("7c553e220d14e1afaa03548a6f0d4652ed7eaa7dbf6c0c1f73a434f5b36fa21d").
    • 1.1.0.3 第二步：继续在shell中执行以下语句：
    %%输入提前定义并且编写好的adapter、procedure、并且严格按照接口文档填写参数列表（公共参数参照前面章节配置）
ebank_adapter_utils:ebank_invoke_procedure({"pochost","ad",[]}).

  • 1.2 模拟前端输入输出

    • 1.2.0.1 打开浏览，访问http://192.168.241.3:4005/admin，登陆，如下图

  • 1.2.0.2 在调试窗口按照前后端接口定义的上送参数填写上送报文

点击提交

如果返回内容如下

{"message":"internal error"}

表示后台程序有问题，请根据后端日志，查找问题，并修改 如果返回内容为正常json，请格式化后查看内容是否与前后端接口文档内容一致，不一致请根据情况修改 cs或者erlang代码

2 2、真实数据调试

真实数据调试的调试方法基本与上面一致，主要分为模拟登陆和不模拟登陆两种

  • 2.0.0.1 模拟登陆：需要根据数据需要修改make_session.conf 配置文件来保证每次调试的数据的正确性。
  • 2.0.0.2 不模拟登陆：需要每次交易前都要通过管理控制台做一遍交易登陆，来保证session中存有交易需要的数据。

注意：两种模式需要使用不同的配置 ebank.conf

%%是否启用模拟session
{login,true}. % true|false|con  代码模拟登陆|报文模拟登陆|完整流程

主要使用true和con两种方式
