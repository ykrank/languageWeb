1 背景

EMP5.1的android框架是在j2me框架的基础上修改的，而j2me框架的一些特点并不适合于android开发，特别是没有充分利用android开发平台的开发特性。其次5.1的android平台在稳定性、适配、运行效率、容错等功能均不够好。

1.1 5.1现有问题列表

  1. 平台架构模块划分不够清晰、封装不够紧密：

    • 解析模块与UI模块耦合太紧；UI模块中包含系统UI、DOM树结构、排版引擎、图片缩放等模块或功能。
    • 模块间经常穿插使用静态对象、方法，破坏模块化结构同时会造成因为强引用存在而导致垃圾未及时回收。

    • 类层次结构、模块接口定义不清。如：Component抽象类中包含众多子类实现。如下列代码，reSetTextViewLayoutParams、mouldTextView等方法都是处理TextView逻辑的。

        public abstract class Component {
      protected void reSetTextViewLayoutParams() {
          TextViewmlpl = (TextView) realView_;
          Paint paint = mlpl.getPaint();
          ...
      }
      /**
       * 依据Css样式，设置TextView类控件的相关属性
       */
      private void mouldTextView() {
          ...
      }
  }

    • 以上问题对与代码阅读理解、bug修改、功能扩展都有很大影响。

  2. 大量逻辑处理封装在UI线程中，这种方式及容易引起“应用程序无响应”。如下述代码中在UI线程先后执行了排版算法（mouldAndLayout（）），LUA脚本，LUA垃圾回收。

  3. 报文布局与系统UI布局掺杂一起逻辑较乱，产生较多bug，不利bug修改且修改对全局影响较大。

  4. 在UI中部分复写的系统方法算法处理不当，容易造成在不同设备上显示效果有差异。如，LPLabel类，自定义方法对字符进行宽、高、折行、绘制等操作；由于算法逻辑封装在onDraw中，同时影响UI线程。
  5. 对UI图片使用自定义方法进行了缩放，影响系统性能而且在缩放过程中容易造成内存溢出。
  6. 现有LuaJava引擎存在有BUG，且异常较多。如，**银行sd卡问题，由于属第三方API不利于BUG修改；另外日志输出不够好；

  7. 在LUA、JS注入的对象中存在大量重复对象，造成内存负担。

 }
    • 上述代码中Component.LUASTATE.pushObjectValue(this);Component.LUASTATE.setGlobal("luascreen");就是向LUA中注入当前LuaScreen对象方法，LUA中即可用luascreen.reflash()访问改方法，但后面又用LUA脚本注入screen对象执行的同样是luascreen.reflash()方法。

  8. 网络引擎在效率与稳定上还有问题，如平凡访问ResponseCode值会出现-1状况或HttpURLConnection.getOutputStream（）时报出bad file number！IO异常。

1.2 针对问题解决方案

在重构中针对1.1描述问题解决方法如下：

  1. 重新划分平台模块，明确模块功能及接口；对类层次接口进行定义，明确其责任及功能；
  2. 引入DOM、Render引擎，以减轻UI模块逻辑释放UI线程，提高程序响应速度。
  3. 增加排版引擎，分离报文布局与系统UI布局，减少相互影响；
  4. 在GUI中尽量使用系统原生API，有利于机型适配，同时也降低不稳定因素。
  5. 去除Bitmap缩放，将此功能转给系统API。同时做好Bitmap优化与内存回收。
  6. 根据平台需求对C++ LUA进行封装，增强日志输出功能；增加LUA引擎透明度，有利于BUG跟踪修改。

  7. 优化LUA、JS注入对象、流程、垃圾回收等方法。

    • LUA实现方式：
      1）对C++ LUA进行jni封装。
      2）使用LuaJava代码示例：

        public class Screen {
      public Screen() {
          Component.LUASTATE.pushObjectValue(this);
          Component.LUASTATE.setGlobal("screen");
      }
      public void refresh(){
      }
  }

    • 上述代码即可注入screen对象，LUA中可直接用screen:refresh()访问

    • JS代码示例：

        public class Screen {
      public Screen() { 
      }
      public void refresh(){
      }
  }
  WebView web = new WebView（）；
  web.addJavascriptInterface(new Screen(), “screen”);

    • 同上所述，JS中即可用screen.refresh()回调到Java中对象方法。

  8. 对HttpURLConnection与HttpClient进行效率与稳定综合对比，选中更优者进行封装。

  9. 做好架构设计、开发文档、规范，并严格执行，做好代码review。
  10. 对报文解析，LUA、JS执行时错误异常进行定位，在调试模式下进行alert提示。

2 系统架构图及说明

EMP Android平台架构如上图所示。
浅蓝色与蓝色为架构内核，由产品负责维护与升级。
绿色为APP与扩展模块，由应用层确定其功能与实现。扩展模块为架构内核提供GUI、LUA、JS扩展接口组成，用以支持应用的定制需求。

2.1 应用层

由html、lua、js等脚本语言编写而成。支持本地应用、互联网应用或两者结合应用。

2.2 EMP框架层

EMP框架又分为数据模块、逻辑模块和扩展模块。数据模块与逻辑模块为产品框架核心；扩展模块为不同应用定制需求提供支持；模块关系见下图。
由于平台特性与browser非常相似，此框架设计参照browser并对其进行适应性的修改与添加。

2.2.1 Render引擎

根据系统信息构建并初始化EMP基本运行环境以及所需数据。
提供EMP平台入口，黏合各功能模块。攘括排版引擎，根据DOM树结构以及样式属性对UI布局进行排版并关联GUI库完成界面绘制。对DOM树结构、属性等进行监听并适时刷新、替换界面。在界面与DOM引擎间进行事件转换、传递。
作为EMP与操作系统连接桥梁，负责EMP与系统特性间方法、数据的匹配与转换。

2.2.1.1 排版引擎

遍历DOM树，依据树结构以及节点属性、样式进行排版。

2.2.2 GUI库

图形用户界面（Graphical User Interface，简称GUI，又称图形用户接口）库，根据报文节点对Android系统图形界面对象进行相应扩展组建GUI库，供Render引擎界面绘制。
实现事件监听，使其支持DOM EVENT接口。从而完成按键事件功能逻辑。
GUI库中界面对象类尽可能使用系统原生API，减少UI类中算法与逻辑处理提高响应效率。
定义GUIFactroy抽象类，提供GUI绘制接口。
定义AndroidGUIFactroy继承GUIFactory类，功能扩展如下：

  1. 实现GUIFactroy抽象类的GUI绘制接口。
  2. 建立UI库与DOM节点关联表，并根据Node名称、类型创建UI。
  3. 完成从节点共有属性、样式、结构到Android 布局的转换。
    如：left、top、right、bottom转换成Android LayoutParams对象中的LayoutParams.x、LayoutParams.y、LayoutParams.width、LayoutParams.height。

  4. 对私有属性，定义如下接口类：

     public interface GUIInterface {
     public void setNodeInfo(Node parent, Node child);
 }

    将节点信息设置入其实现者，完成节点属性key与value到Android View属性方法转换。如：

     public InputBox extends EidtText implements GUIInterface{
     private Element mElement = null;
     public void setNodeInfo(Node parent, Node child){
         mElement = (Element) child;    
         String value = element.getAttribute(“value”);
         setText(value);
     }
     @Override
     protected void onTextChanged(CharSequence text, int start, int lengthBefore, int lengthAfter) {
         mNode.setAttribute(“value”, text);
     }
     ...
 }

实现GUI扩展接口，以对DOM树增加节点提供支持。

GUI时序图说明如下：

  1. AndroidGUIFactoryImp根据Node信息，创建InputBox并将公共属性转换成LayoutParams设入InputBox。
  2. AndroidGUIFactoryImp将Node通过GUIInterface设入InputBox，InputBox读取Node信息完成私有属性设置。AndroidGUIFactoryImp再根据Node父节点，将InputBox加入其父ViewGroup中完成显示。
  3. 用户点击屏幕，向InputBox输入文本内容，InputBox监听到文本改变后通过onTextChanged将改变后文本添加到Node。

2.2.3 DOM引擎

文档对象模型（Document Object Model简称DOM），集合报文DOM树（见图）、EMP功能对象库两大模块。
作为报文、LUA、JS等与系统、视图之间的连接桥梁。

2.2.3.1 DOM树

根据报文中节点属性以及等级关系创建DOM树（如上图）结构。支持通过DOM访问DOM树中每个节点，并可以添加、移除、改变或重构整颗DOM树。

2.2.3.2 EMP功能对象库

创建功能类库，给LUA、JS等脚本获取系统信息、访问DOM树等等提供支持。
具体类实现如：

  • Document对象，每个载入EMP的报文都会成为Document对象。Document 对象使我们可以从脚本中对报文页面中的所有元素进行访问，如Document.getElementsByName()返回与指定的所有属性匹配的所有控件对象。
  • Screen对象，对系统屏幕信息反馈。如Screen.width返回屏幕宽。Screen.height返回屏幕高。

等等具体对象、方法可参考HTML DOM相关文档。

2.2.3.3 LUA 引擎

LUA引擎分黏合层、JNI层、LUA内核层三部分。

  1. 黏合层，实现引擎初始化以及LUA扩展接口。负责注入基础LUA对象、方法，连接DOM模块，完成LUA对平台、系统等功能操作。具体方法封装参见EMP Lua Library；
  2. JNI层，完成基本LUA的本地封装，连接LUA so库；提供完善的LUA日志输出方法；给黏合层提供支持。
    定义错误提示提示接口，实现LUA执行alert错误提示。
  3. LUA内核层，采用LUA5.1.4版本库。
2.2.3.4 Webkit引擎

Webkit引擎分JS模块与WebView模块。

  1. JS模块，创建JS运行基本对象库并提供JS扩展接口。黏合DOM引擎完成对象库具体功能。
  2. WebView模块， 负责初始化WebView构建JS运行环境、提供扩展接口、加载JS对象库。

2.2.4 EMP扩展

作为EMP android平台框架的扩展，由GUI、LUA、JS等模块提供扩展接口组成。实现在框架中没有定义或超出定义的控件以及功能逻辑模块。

2.2.5 数据引擎

对框架所需数据进行统一封装，分为本地数据与网络数据两种类型；并负责连接安全引擎实现适时对数据进行加解密。

2.2.5.1 本地数据

根据Android存储方式性能比对，较大数据量读写操作如“图片”等，文件系统方式远比数据库方式快；但数据结构较复杂时，文件系统操作复杂且效率不高；因此综合考虑本地数据存储方式如下：

  1. 离线报文、图片等数据采用文件系统保存。对使用频繁且较小的数据保存于系统默认目录；
  2. 简单数据结构，比如设置类信息采用SharedPreferences方式存储；
  3. 复杂数据结构，且有查询、排序等操作的采用数据库方式存储。

以上方式均得考虑对过期数据进行删除清理。
提供sd卡、数据库、文件读写接口，以便LUA、JS进行访问。

2.2.5.2 网络数据

HttpURLConnection与HttpClient对比见下表，实现HttpClient引擎封装，使其支持WIFI、NET、WAP等网络连接方式；异步与同步网络处理流程；GET与POST请求类型。

URLConnection HTTPClient
