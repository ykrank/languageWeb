EMP3.2 & EMP5.2 双引擎继承

本文档针对客户端3.2版本与EMP5.2版本的双引擎实现.

双引擎方案的实现思路: 将EMP5.2 客户端引擎嵌入到3.2架构中. 对于以3.2方式开发的业务使用老框架引擎. 新增的业务可以以EMP5.2方式实现,并通过EMP5.2引擎处理相关内容.

引擎的切换处理方案: 通过channel标签属性通知客户端某交易使用的引擎. 比如在channel标签中加入new_engine="true"属性通知客户端此交易使用EMP5.2引擎. 如果需要从EMP5.2切换会3.2.可以通过定制lua接口实现.

共享加密信道功能.由于后台服务依然要使用同一个服务器.因此对同一个用户依然需要使用相同的加密信道和session. 需要实现信道与seesion共享.

1 实现方案

1.1 文件名及类名修改

由于objective-c 不支持包和namespace. 因此要求所有的类及函数不能重名.
我们需要将3.2及EMP5.2中相同的类及函数做重命名处理.
整理了部分需要重命名的类如下,其中部分类为工具类.可以直接使用EMP5.2中的.
可以通用的类:AES,Base64,DES,HMac,RSA,URICode,Utils,ZNLog,hsmcli.
建议使用EMP5.2的类:DataBaseManager,TLS. 这部分类在EMP5.2中包含了3.2的功能,并且功能更强大.
需要修改类名的类:Config,HTTPConnection,UIAlert.appSign

1.2 函数名修改

部分C的函数也存在重名的问题. 需要修改.

  • 将EMP提供的Framework 添加到工程.

3 实现引擎切换功能

  • 初始化EMP5.2引擎.

      - (void)initEMPExtension{
      if (self.empRootController == nil) {
          UIContentViewController *empRootController = [[UIContentViewController alloc] init];
          self.empRootController = empRootController;
          [empRootController release];
      }
      if (self.channelRequest == nil) {
          ChannelRequest *channelRequest = [[ChannelRequest alloc] init];
          self.channelRequest = channelRequest;
          [channelRequest release];
      }
  }

    ChannelRequest 为封装的请求及返回报文处理类.此类的功能是以EMP3.2引擎的方式向服务器发送请求,并将返回的报文以EMP5.2引擎解析并执行.

    3.1 实现两个引擎的切换

    3.1.1 实现引擎切换功能函数:

      - (void)addOldView{
      // 切换到老框架引擎
      self.window.rootViewController = nil;

      [self.window addSubview:displayWindow_];
      [self.window addSubview:mainMenuScrollSuperView_];
      [self.window addSubview:keyboardCancelView_];
      [self.window addSubview:datePickerView_];
      [self.window addSubview:popViewBgView_];
  }
  - (void)removeOldView{
      // 删除老框架引擎
      // 此方法根据具体项目中加载的视图
      [displayWindow_ removeFromSuperview];
      [mainMenuScrollSuperView_ removeFromSuperview];
      [keyboardCancelView_ removeFromSuperview];
      [datePickerView_ removeFromSuperview];
      [popViewBgView_ removeFromSuperview];
  }
  - (void)addEMPView{
      // 切换到EMP5.2引擎
      [self initEMPExtension];
      // 加载EMP程序,移除旧框架视图
      [self removeOldView];
      self.window.rootViewController = self.empRootController;
  }

3.1.2 解析服务器返回的引擎标示

服务器可以在channel节点中加入属性new_engine来表示此也许需要使用的引擎.如当new_engine='true'时,表示使用EMP5.2.

根据EMP3.2的实现逻辑,对于二级菜单的解析操作在ChannelListParser的- (void) parseStartElement: (NSXMLParser *)parser didStartElement:(NSString *)elementName namespaceURI:(NSString *)namespaceURI qualifiedName:(NSString *)qName attributes:(NSDictionary *)attributeDict方法中. 在此方法的[elementName isEqualToString:@"collection"]情况下会解析二级菜单. 再此if的block中加入如下代码,注意这里需要为Channel类加入属性new_engine.

 NSString *new_engine = [attributeDict objectForKey:@"new_engine"];
 if ([new_engine isEqualToString:@"true"]) {
       channel.new_engine = YES;
 }

4 其他问题

4.1 TLS

为了实现加密信道的共享,将EMP3.2中使用的加密信道替换为EMP5.2的信道实现.

  1. 新框架采用的证书签名长度未256位,而在EMP3.2项目中使用的是128位.因此需要更新证书.如果必须使用128位的签名长度证书.可以修改ROOT_CER_FILE_SIGN_DATA_LENGTH为128.

  2. 关于加密算法的修改.EMP3.2网络请求的加密算法在LightPoleAppDelegate类中.将其实现改为采用EMP5.2的加密算法. 修改LightPoleAppDelegate类中的- (NSData *) AES256EncyrptBody:(NSData *)body,- (NSData *) AES256DecyrptBody:(NSData *)body 使用产品TLS中的加密方法.

     - (NSData *) AES256EncyrptBody:(NSData *)body {
     return [[TLS sharedTLS] AES256EncyrptBody:body];
 }

 - (NSData *) AES256DecyrptBody:(NSData *)body {
     return [[TLS sharedTLS] AES256DecyrptBody:body];
 }

  3. 修改- (NSString *) getCookie方法.EMP3.2中的session是在tis后添加的,新框架没有此操作.

     - (NSString *) getCookie 
 {
     NSString *result = [[self getKeyInfo] objectForKey:TAG_PROPERTY_COOKIE];
     if (!result) {
         NSString *sID = [TLS sharedTLS].sessionId;
         if (sID) {
             NSMutableString *sessionID = [[NSMutableString alloc] init];
             [sessionID appendString:@"_session_id="];
             [sessionID appendFormat:@"%@",sID];
             [[self getKeyInfo] setValue:sessionID forKey:TAG_PROPERTY_COOKIE];
             [sessionID release];
             result = [[self getKeyInfo] objectForKey:TAG_PROPERTY_COOKIE];
         }
     }
     return result;
 }
  4. 使用到了部分新产品TLS中未开放的接口.如tls.clientKey,tls.clientIv,建议采用其他标示.

4.2 数据库

老框架与新框架使用了相同的数据库文件 sqldatabase.sql. 其中新框架中增加了一张tls的表. 另外一张表结构相同. 因此这张表可以合并. 因此这里建议使用EMP5.2中的数据库.
