WP 离线2.0更新

离线2.0更新 更新内容：

  1. hash接口字段添加
  2. 离线下载方式升级
  3. 预置资源加密
  4. lua接口优化

1 hash接口字段添加

2.0之前只返回m字段，后添加了更多的信息，因此封装在了json字符中，包括：

t：tcp下载的端口号
st：离线资源下载方式 (0，1，2，3）具体查看后台提供的下载文档
h：离线资源更新地址 （由于1.0升级 2.0版本路径可能发生变化）
m：离线资源更新标志 （0，1，2，3）

程序在DatabaseManager.V2_0UpdateHash(string json)中统一进行了处理.

2 离线下载方式升级

离线资源下载有HTTP和TCP两种，每种又有四种分类，具体看文档，下载逻辑入口都在RYTOfflineStorageManager类的V1DownCommonOrOptionalFile方法和V1EnsureDownloadFile方法进行处理，后者指用于确保更新（只支持TCP），下面介绍程序对每种下载方式的处理。

HTTP：

ST == 0或3：HTTP明文 1：HTTP信道明文 2：HTTP信道密文 3：HTTPS 离线2.0之后，不在拼接url，而是将路径post到服务器，而wp没有专门的接口处理HTTPS，只要手动将服务器的证书安装到手机上，程序自动会进行加密。因此当0和3时用PostClient对象进行下载。

ST == 1或2：直接调用旧的信道封装类HttpRequest，并在构造函数里设置是否加密即可。

TCP：

TCP通过SocketClient进行数据下载，并根据ST的不同.ST == 0：TCP明文 1：TCP信道明文 2：TCP信道密文 3：SSL，调用不同的内部类对参数以及下载数据进行处理。调用时只要将ST参数传入构造函数即可初始化相应的数据处理对象DataToolV2,DataToolV2TLS,前者用于非信道的数据处理，后者用于信道数据，他们都继承于DataToolBase。

ST == 0或3：使用DataToolV2。
ST == 1或2：使用DataToolV2TLS，并根据是否需要加密进行初始化。

需要注意的是
1.HTTPS下载只有客户端验证服务器证书通过之后（客户端通过浏览器进行手动安装），才能正常下载。 2.SSL下载时SocketClient调用ErtRTComponent库提供的RYTSSLSocket下载数据。RYTSSLSocket通过调用ErtLib中的SSLSocket，和国密一样调取openssl库赖实现。ErtRTComponent只是一个简单的数据传递。

3 预置资源加密

预置资源加密的详细信息请查看：wp的资源加密类

4 接口添加

离线2.0一共添加了三个接口，这三个接口只能用于2.0及更高版本，他们分别为：

  1. offline:update_resource(params)
  2. offline:checkOfflineFile(filename)
  3. offline:checkOfflineFileWithServer(filename)

其中第一个offline:update_resource(params)添加了资源下载的回掉，当离线资源下载时将已下载和全部下载的数据量实时反馈给lua，并在最后完成或失败结束时回掉结果。 程序在 RYTOfflineStorageManager 类中添加了两个Action processAction和finishAction，程序进入后进行绑定。如想查看具体执行情况，可直接查看程序入口报文：main.xml.

WP 5.2 离线优化前后使用对比

WP EMP5.2 添加了离线协议版本（Offline_Version), 0即为旧版本，而1相对于版本0对文件的查找使用结构进行了大量修改，配合新扩展的Lua接口，下面就从对比的角度介绍下新版离线协议。

1 离线协议主体介绍

有关离线文件处理主要分为三大块：

  1. RYTong.MainProject.Controller 命名空间下的RYTOfflineStorageManager类，负责主要的离线文件的管理。
  2. RYTong.LuaScript 命名空间下的LuaOffline类，负责提供离线文件操作的Lua接口。
  3. RYTong.DataBaseLib 命名空间下的DataBaseManager类负责提供几乎所有IO基础操作。

1，2两个模块的修改基本是新版本替换旧版本的方法，只能在相应的版本中使用，而3中除了添加新版本的方法外，有一些读取和判断的方法新旧版本可共用。

2 离线配置与预置资源的处理

离线资源版本的配置在主工程的配置文件ConfigManager类中 Offline_Version（0/1）进行设置。 为了方便测试，进入程序前可以对离线版本进行更改（为保证程序正常测试，如果预置资源位1时不可修改）

RYTOfflineStorageManager.cs （左侧为新，右侧为旧）

程序真正开始运行时首先会初始化离线资源描述，如果是第一次进入会将预置资源导入独立存储区：
V1CheckFirstEmbedResourceOffline() == CheckFirstEmbedResourceOffline()

由于1版本需要对0版本的兼容，因此，当客户端升级离线资源到1版本时，客户端独立存储区中也有可能已经下载了离线资源，这时上述方法会根据规定的处理逻辑将部分0版本的资源升级为1版本，并且与预置资源的内容融合在一起，而开发人员在打包时只需要将配置文件 Offline_Version 修改为1版本即可。具体处理逻辑可查看： 客户端设计文档

3 离线文件的更新

RYTOfflineStorageManager.cs中添加的新方法大多都是为扩展新版的Lua接口使用，离线资源的客户端开发文档中对新旧接口有详细的介绍： 客户端设计文档

新版离线文件更新操作

新版离线更新请求，会先去与服务器校验本地描述文件的Hash值，获取更新标志并返给Lua，通过更新标志再去判断是否要进行更新描述文件，获取描述文件后依然由Lua来判断是否进行更新操作，整个更新流程控制权基本移交给Lua进行控制下面看下模块1中的修改：

通过上传已本地hash获取更新标志：
OfflineUpdate_Hash(Action action) == 无

根据更新标志更新描述文件
V1ClientUpdateAction() == ClientUpdateAction()

更新离线文件
V1UpdateOfflineResource() == JustUpdateOfflineResource()

根据参数下载具体文件(包括可选资源)
DownloadOfflineFile() == DownloadWithWebClient()

离线Lua接口修改

获取可选文件的描述信息
GetOptNameRevDesc() == GetServerDownloadOptJson()/GetClientDownloadOptJson()

校验文件是否可用
V1CheckOfflineFileAvailable() == CheckOfflineFileAvailable()

根据插件名删除插件资源
V1RemoveOptionalFile() == 无

根据文件名删除资源
V1RemoveOfflineFile() == RemoveClientOfflineFile()

DatabaseManager.cs 由于程序离线描述存储结构以及查找方式的改变，因此只要方法的签名中带有旧类的方法在新版本中都将废弃

如：

GetAllFileNames_New()
ReadFileDesList()
ParserResourceDesc()
SaveResourceDescFile()

修改

根据文件名和路径校验文件是否可用并读取返回
V1CheckOfflineFileAvailable() == CheckOfflineFileOrPlugInFile() V1ServerJsonToOfflineObj() == ParserDescData()
